# 1. 工作台目标

工作台不是一个“日志后台”，而是一个 **问题定位操作台**。

它要解决的不是“看到了很多数据”，而是这 3 件事：

1. **快速选中问题点**
2. **沿链路看清问题传播过程**
3. **拿到可执行结论**

所以工作台信息架构建议按这三个认知阶段来设计：

```text
发现问题
→ 看链路
→ 出结论
```

对应成页面能力就是：

```text
入口定位
→ 证据浏览
→ 诊断结论
```

# 3. 页面分层设计

我建议工作台分成 **4 层信息结构**：

---

## L1：全局上下文层
解决“我当前在看哪一个应用、哪一个页面、哪一个会话”。

包含：
- 应用名
- 环境（dev/test/prod）
- 页面 URL / route
- session_id
- 时间范围
- 当前模式（Inspect / Trace / Diagnosis）

---

## L2：问题入口层
解决“我从哪里开始看问题”。

包含两种入口：

### 入口 A：从交互开始
- 最近点击列表
- 最近异常交互
- 搜索某个 interaction

### 入口 B：从页面值开始
- 最近 inspect 目标
- 当前页面已标注异常点
- 指定 DOM / 组件搜索

---

## L3：链路分析层
这是核心区，解决“链路怎么传递”。

包含：
- Trace 时间线
- Reverse Lineage 反向溯源
- Forward Lineage 正向主链
- Domino 因果链图
- 节点证据详情

---

## L4：结论输出层
解决“最终原因是什么，要怎么改”。

包含：
- 根因排序
- 证据链摘要
- 修复建议
- AI 解释
- 导出/复制报告

# 6. 主工作台页的信息架构

这个页面最关键。  
我建议拆成 **6 个功能区块**。

---

## 6.1 顶部全局栏

### 模块
- 应用切换
- 环境切换
- 页面 route
- session 选择
- 时间范围
- 搜索框
- 模式切换

### 模式切换建议
```text
[Inspect 反向溯源]
[Trace 正向链路]
[Diagnosis 诊断视图]
```

### 作用
让同一页面根据用户任务切换视角，而不是跳一堆页。

---

## 6.2 左侧入口面板

这是“问题入口区”。

### 区块 A：最近交互
显示：
- interaction 类型
- 目标元素
- 时间
- 状态
- 是否异常

### 区块 B：最近 Inspect 目标
显示：
- DOM selector
- 当前文本
- 组件名
- 是否已有诊断结果

### 区块 C：异常筛选
标签建议：
- 显示为空
- 显示 `--`
- 点击无响应
- 接口成功但页面未更新
- 组件未刷新
- DOM 不可见

### 区块 D：组件目录
显示当前 session 中活跃组件，方便按组件切入。

---

## 6.3 中央分析区

中央区是主战场。  
建议做成 **Tab + 主画布** 结构。

### Tab 1：Trace
适合“点击后发生了什么”。

内容：
- 事件时间线
- 节点流
- 成功/失败状态
- 链路中断点

### Tab 2：Lineage
适合“这个值从哪来”。

内容：
- 反向血缘链
- 每层值摘要
- 每层证据

### Tab 3：Domino
适合展示核心因果传播过程。

内容：
- 多米诺骨牌式节点流
- 每个节点代表一个事件/状态变化
- 高亮异常中断点或错误传播点

### Tab 4：Evidence
适合开发深挖。

内容：
- API response
- store snapshot
- render props
- DOM before/after
- raw events

---

## 6.4 右侧诊断结论区

这个区域永远固定存在，不随 Tab 消失。  
因为用户最终就是要结论。

### 模块 1：诊断摘要
一行话总结当前问题。

例：
- 更可能是前端 formatter 问题
- 更可能是组件未响应 store 更新
- 更可能是接口字段缺失

### 模块 2：根因排序
列表展示：
- 根因名称
- 置信度
- 受影响层级

### 模块 3：证据摘要
展示关键 evidence：
- API 字段值
- store 值
- render 输出
- DOM 状态

### 模块 4：修复建议
展示：
- 建议改代码点
- 建议补充测试点
- 建议增加监控点

### 模块 5：AI 解释
可展开：
- 面向开发版本
- 面向测试版本
- 面向产品版本

---

## 6.5 底部证据详情区

这里不要抢主视觉，但非常重要。  
适合做成可展开区域。

### Tab 建议
- Raw Events
- Request / Response
- Store Before / After
- Render Props / State
- DOM Before / After
- Metadata

### 作用
当用户对上面结论不信时，可以下钻验证。

---

## 6.6 节点详情抽屉

在中央区点任何节点，都打开一个侧边抽屉或浮层。

### 节点详情建议字段
- 节点类型
- 时间戳
- 所属 interaction
- 组件名
- 请求接口
- 输入值 / 输出值
- 父子节点
- 关联 evidence
- 命中规则

这个对工程使用非常关键。

# 7. 三种核心视图设计

我建议工作台主工作区围绕 3 种视图构建。

---

## 7.1 Trace 视图
适合回答：

- 点击后请求有没有发
- 请求返回后有没有更新 store
- 哪一步断了

### 推荐结构
```text
时间轴 + 节点流
```

### 展示项
- click
- handler
- request
- response
- state_update
- render
- dom_update

---

## 7.2 Lineage 视图
适合回答：

- 这个 `--` 从哪里来的
- 当前显示值对应哪个 store key / api field
- 是接口错，还是转换错

### 推荐结构
```text
逆向分层链
DOM
↑
Render
↑
Component/Prop
↑
Store/Selector
↑
API Field
```

### 每层展示
- 来源键
- 当前值
- 证据链接
- 是否异常

---

## 7.3 Domino 视图
适合做产品差异化。

### 推荐结构
把每个关键节点做成“骨牌”：

```text
[Click]
   ↓
[Request]
   ↓
[Response amount=0]
   ↓
[Formatter output=--]   <-- 高亮异常
   ↓
[DOM text=--]
```

### 好处
- 比传统 trace 更直观
- 非研发也更容易看懂
- 很适合演示

### MVP 做法建议
先做 **静态流向 + 节点状态高亮**，  
后续再做动画倒塌效果。

# 8. 按用户角色的信息架构差异

虽然先做一套页面，但展示重点可按角色切换。

---

## 开发视角
重点看：
- 组件
- store
- render
- formatter
- 原始证据

## 测试视角
重点看：
- 页面异常现象
- 输入输出不一致
- 接口返回 vs 页面展示差异
- 回归建议

## 产品视角
重点看：
- 用户看到什么异常
- 哪一层出问题
- 影响范围
- 当前建议动作

### 实现方式建议
右侧 AI 解释区增加切换：
```text
[开发] [测试] [产品]
```

# 14. 一句话总结

这套工作台信息架构的核心是：

> **左边负责“从哪里进”，中间负责“链路怎么看”，右边负责“最后怎么判”。**

也就是：

> **入口 → 链路 → 结论**

这正好对应你产品的核心价值，不会退化成一个只会堆日志的后台。
