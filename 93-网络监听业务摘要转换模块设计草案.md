下面给你一版 **“网络监听 → 业务摘要转换” 的前端模块设计草案**。  
我按你现在的路线来收口：

- 浏览器插件为主
- 尽量低侵入
- 保留原始网络事件
- 叠加一层业务调试友好的摘要
- 优先服务 **时间线展示 / 问题定位 / diagnosis 输入**

---

# 1. 模块目标

这个模块的目标不是替代 DevTools，也不是完整业务建模，而是：

> 把浏览器插件监听到的原始网络事件，转换成用户更容易理解、对诊断更有帮助的“业务摘要事件”。

它主要解决 3 个问题：

1. **原始请求过于技术化**
   - 用户看不懂 path / method / status / payload

2. **网络事件和业务动作之间缺少解释层**
   - “这个请求到底是在干嘛”

3. **需要更容易挂到诊断时间线和 domino chain 上**
   - “点了查询 -> 查订单列表 -> 成功但为空 -> 页面没展示”

---

# 2. 设计原则

---

## 2.1 保留原始事件，摘要是派生层
不能只保留转换结果。必须同时保留：

- raw network event
- normalized network event
- business insight summary

因为后续：
- UI 展示用摘要
- 诊断规则可能仍要用原始字段
- 调试时要能回看技术细节

---

## 2.2 摘要是“提示”，不是“真相”
这层应该输出：
- `actionLabel`
- `resultCategory`
- `debugHints`

而不是直接输出强结论：
- “后端错了”
- “前端有 bug”

结论应留给后端 diagnosis 或更高层规则。

---

## 2.3 桥接语义优先于 URL 猜测
如果页面桥接已经告诉你：

- `businessAction = submitOrder`
- `module = order-list`

那要优先使用这些信息，  
不要再强依赖 URL path 猜业务。

---

## 2.4 只做轻量、安全的响应摘要
只提取必要信息：
- success/code/message
- total/count/items.length
- 空结果
- 缺失关键字段

不要默认长期保留完整 body。

---

# 3. 模块在前端链路中的位置

推荐链路：

```text
network observer
  -> raw network event
  -> sanitizer
  -> normalizer
  -> business insight transformer
  -> timeline item builder
  -> session recorder
  -> upload / local preview
```

其中我们这次设计的重点是：

- `normalizer`
- `business insight transformer`
- `timeline item builder`

# 5. 各模块职责

---

## 5.1 `network-observer.ts`
**职责：**
- hook `fetch`
- hook `XMLHttpRequest`
- 采集原始请求/响应事件

**输出：**
- `RawNetworkEvent`

这是数据源头，不做业务解释。

---

## 5.2 `network-sanitizer.ts`
**职责：**
- URL 脱敏
- query 参数过滤
- body 摘要裁剪
- 敏感字段过滤
- response 摘要脱敏

**输出：**
- `SanitizedNetworkEvent`

作用是保证后面的标准化和展示不会直接暴露敏感数据。

---

## 5.3 `network-normalizer.ts`
**职责：**
把原始浏览器层差异统一成标准结构：

- fetch / xhr 统一
- duration 统一
- method / path / status 统一
- requestId / traceId 统一
- response 摘要统一

**输出：**
- `NormalizedNetworkEvent`

---

## 5.4 `mapping-registry.ts`
**职责：**
提供“接口 → 业务动作”映射规则注册与匹配能力。

例如：
- `/api/orders` -> 查询订单列表
- `/api/orders/:id` -> 获取订单详情
- `/api/inventory/check` -> 校验库存

**输出：**
- `ApiBusinessMapping | undefined`

---

## 5.5 `default-mappings.ts`
**职责：**
内置通用映射规则。

比如基于 method + REST path 的默认猜测：

- `GET + /xxx/list` -> 查询列表
- `GET + /xxx/:id` -> 获取详情
- `POST + /xxx` -> 新建
- `PUT/PATCH + /xxx/:id` -> 更新
- `DELETE + /xxx/:id` -> 删除

这层解决“没有业务自定义配置时也能用”。

---

## 5.6 `url-pattern.ts`
**职责：**
- URL 规范化
- path pattern 提取
- resource name 提取
- route token 分析

比如：
- `/api/orders/123/detail?x=1`
  -> `/api/orders/:id/detail`

---

## 5.7 `response-summary.ts`
**职责：**
从响应中提取轻量摘要：

- 是否成功
- code/message
- 是否空
- list 长度
- total/count
- 是否缺少关键字段

这层不要直接做业务结论，只做结构化摘要。

---

## 5.8 `network-insight-transformer.ts`
**职责：**
核心模块。把 `NormalizedNetworkEvent` 转成 `NetworkInsight`。

输入：
- network event
- 当前页面上下文
- 最近 bridge interaction / module context
- mapping registry

输出：
- `NetworkInsight`

---

## 5.9 `network-timeline-builder.ts`
**职责：**
把 `NetworkInsight` 转成适合 UI 展示和 session 时间线的简短条目。

例如：
- `查询订单列表`
- `订单列表返回为空`
- `创建订单失败：504 超时`

---

## 5.10 `types.ts`
**职责：**
统一定义整个模块涉及的数据类型。

---

# 6. 类型设计草案

---

```ts
export interface RawNetworkEvent {
  source: 'fetch' | 'xhr';
  phase: 'request' | 'response' | 'error';
  requestId: string;
  traceId?: string;
  method: string;
  url: string;
  status?: number;
  startedAt: number;
  endedAt?: number;
  durationMs?: number;
  requestHeaders?: Record<string, string>;
  requestBody?: unknown;
  responseHeaders?: Record<string, string>;
  responseBody?: unknown;
  errorMessage?: string;
}
```

---

## 6.2 脱敏后事件

# 7. 转换流程设计

---

## 步骤 1：采集原始事件
`network-observer` 监听请求响应，得到 `RawNetworkEvent`

---

## 步骤 2：脱敏
`network-sanitizer` 处理：
- query 白名单
- body keys 提取
- response 提取 count / total / code / message
- 敏感字段 masking

输出 `SanitizedNetworkEvent`

---

## 步骤 3：标准化
`network-normalizer` 统一：
- method 大写
- urlPattern
- actionKind
- statusCategory
- resourceName
- requestSummary
- responseSummary

输出 `NormalizedNetworkEvent`

---

## 步骤 4：业务映射匹配
`mapping-registry` 尝试匹配：
1. bridge businessAction 关联规则
2. 自定义映射规则
3. 默认 REST 风格映射

---

## 步骤 5：生成业务摘要
`network-insight-transformer` 产出：
- actionLabel
- requestText
- responseText
- resultCategory
- debugHints

---

## 步骤 6：生成时间线项
`network-timeline-builder` 生成适合 SidePanel / 工作台预览的文案条目。

---

# 8. 关键转换策略

---

## 8.1 动作识别优先级
推荐优先级如下：

```text
页面桥接 businessAction
  > 映射表命中
  > method + urlPattern 默认猜测
```

---

## 8.2 模块识别优先级
推荐优先级如下：

```text
最近 bridge module state
  > pageContext.module
  > mapping.module
  > undefined
```

---

## 8.3 结果分类规则
建议先做简单稳定的规则：

### timeout
- errorMessage 包含 timeout
- duration 超过阈值且无响应

### failed
- status >= 400
- responseSummary.success === false
- responseSummary.hasErrorField === true

### empty
- status success
- `listCount === 0`
- 或 `isEmpty === true`

### invalid_data
- success 但关键结构字段缺失
- 例如预期 list 接口却没有 list/data

### success
- status 2xx
- 且非 empty / invalid_data

# 9. 示例规则

---

## 9.1 默认映射规则示例

```ts
export const defaultMappings: ApiBusinessMapping[] = [
  {
    id: 'get-list',
    match: { method: 'GET', pathRegex: /\/api\/.+(list|items|query|search)/i },
    actionKey: 'queryList',
    actionLabel: '查询列表',
  },
  {
    id: 'get-detail',
    match: { method: 'GET', pathRegex: /\/api\/.+\/[^/]+$/i },
    actionKey: 'queryDetail',
    actionLabel: '查询详情',
  },
  {
    id: 'post-create',
    match: { method: 'POST', pathRegex: /\/api\/.+/i },
    actionKey: 'createResource',
    actionLabel: '提交创建',
  },
];
```

---

## 9.2 业务自定义规则示例

```ts
export const orderMappings: ApiBusinessMapping[] = [
  {
    id: 'query-orders',
    match: {
      method: 'GET',
      urlPattern: '/api/orders',
    },
    actionKey: 'queryOrders',
    actionLabel: '查询订单列表',
    module: 'order-list',
    buildRequestText: ({ event }) => {
      const keys = event.requestSummary?.queryKeys?.join(', ');
      return keys
        ? `查询订单列表（筛选条件: ${keys}）`
        : '查询订单列表';
    },
    buildResponseText: ({ event }) => {
      const count = event.responseSummary?.listCount;
      if (count === 0) return '接口成功返回，但结果为空';
      if (typeof count === 'number') return `接口成功返回 ${count} 条记录`;
      return '接口请求成功';
    },
    buildHints: ({ event }) => {
      if (event.responseSummary?.listCount === 0) {
        return ['请确认筛选条件是否过严', '页面可能展示为空列表'];
      }
      return [];
    },
  },
];
```

# 12. 与诊断会话的集成方式

建议在 session 里同时记录两类对象：

---

## 原始/标准化网络事件
用于后端诊断和技术追溯

```ts
{
  type: 'network.response',
  payload: normalizedEvent
}
```

---

## 业务摘要事件
用于 UI 预览和更强可读性

```ts
{
  type: 'network.insight',
  payload: insight
}
```

这样最好，不冲突。

---

# 13. MVP 推荐最小实现范围

如果你想控制复杂度，我建议先只做：

### 必做
- fetch/xhr 监听
- sanitizer
- normalizer
- method + urlPattern 默认映射
- success / failed / empty / timeout 分类
- 时间线摘要文案

### 推荐增强
- 业务映射表注册
- bridge businessAction 优先
- response listCount / total 提取
- debugHints 生成

### 延后
- 复杂 schema 推断
- 自动字段学习
- 多接口聚合成事务级摘要
- 智能推断强业务语义

---

# 14. 最终建议

这套模块我建议你正式命名成：

> **Network Insight Transformer**

它不是简单“网络日志展示”，而是把网络层信号转换成：

- 用户可读
- 调试友好
- 可进入时间线
- 可服务后端诊断

的中间层。
