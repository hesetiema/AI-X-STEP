下面给你一版 **前端信息架构 / 页面草图说明（Popup + SidePanel + 工作台入口）**。  
我按你现在的 MVP 路线来收口：

- **浏览器插件是主入口**
- **Popup 负责最短操作闭环**
- **SidePanel 负责会话内诊断辅助**
- **工作台负责正式结果消费**

我先不画高保真 UI，而是给你 **信息架构 + 低保真页面草图说明**，方便你后面继续出 wireframe。

---

# 1. 前端信息架构总览

可以把整个前端体验分成 3 个面：

```text
A. Popup
   - 快速开始/结束诊断
   - 查看当前录制状态
   - 提交 session
   - 跳转工作台结果

B. SidePanel
   - 录制中的会话详情
   - 事件时间线预览
   - 页面上下文
   - 用户备注 / 预期 vs 实际
   - 提交前检查

C. Diagnosis Workstation
   - 正式诊断结果展示
   - top cause / supporting causes / symptoms
   - domino chain
   - evidence drill-down
```

---

# 2. 用户主路径

建议你把主路径定义得非常清晰：

## 2.1 最短闭环
1. 用户打开业务页面
2. 点击插件 Popup
3. 点击“开始诊断”
4. 复现问题
5. 点击“结束并提交”
6. 查看提交成功状态
7. 跳转工作台详情页

这是 **最核心 MVP 闭环**。

---

## 2.2 增强闭环
1. 用户开始诊断
2. 在 SidePanel 中实时看采集摘要
3. 补充问题描述 / 预期结果 / 实际结果
4. 提交
5. 工作台查看根因与 domino chain

这是 **下一层增强体验**。

---

# 3. 信息架构分层

---

## 3.1 Popup 定位
Popup 不是复杂工作台，只做 **“快进快出”**：

- 当前页可不可以诊断
- 现在是不是在录制
- 当前录了多少
- 一键结束并提交
- 提交后去工作台

也就是说，Popup 是 **控制台**，不是 **分析台**。

---

## 3.2 SidePanel 定位
SidePanel 是 **录制过程中的辅助视图**：

- 当前诊断会话概览
- 已采集事件流
- 页面与模块上下文
- 用户补充说明

它帮助用户确认：
- “我录到了”
- “录得对不对”
- “提交前是否需要补充语义”

---

## 3.3 Workstation 定位
工作台是正式诊断结果承载页：

- 结果解释
- domino chain
- evidence drill-down
- 二次查看与复盘

它不是插件的一部分，但插件需要把用户顺畅送到这里。

---

# 4. Popup 信息架构

---

## 4.1 Popup 页面结构
建议 Popup 只分 5 个区块：

1. 顶部状态区
2. 当前页面识别区
3. 录制控制区
4. 会话摘要区
5. 结果/跳转区

---

## 4.2 Popup 草图说明

```text
┌────────────────────────────┐
│ 诊断助手                    │
│ 当前状态：未录制 / 录制中    │
├────────────────────────────┤
│ 当前页面                     │
│ 应用：ERP Console           │
│ 模块：订单                  │
│ 页面：/orders/create        │
│ 状态：可诊断 / 未识别       │
├────────────────────────────┤
│ [开始诊断]                  │
│ [结束并提交]                │
├────────────────────────────┤
│ 本次会话摘要                 │
│ 交互事件：12                │
│ 请求事件：5                 │
│ 错误事件：1                 │
│ 桥接事件：3                 │
├────────────────────────────┤
│ 最近结果                     │
│ 已生成诊断 #diag_xxx        │
│ [打开工作台]                │
└────────────────────────────┘
```

---

## 4.3 Popup 区块职责

### A. 顶部状态区
展示：
- 产品名/插件名
- 当前录制状态：
  - 未录制
  - 录制中
  - 已停止待提交
  - 上传中
  - 上传成功
  - 上传失败

作用：
- 用户一眼判断当前插件状态

---

### B. 当前页面识别区
展示：
- 当前页面是否支持诊断
- 当前页面上下文
  - app
  - module
  - route/url
- 是否读到 bridge context

作用：
- 告诉用户插件是否“看懂了当前页”

---

### C. 录制控制区
核心按钮建议只有两个主动作：

#### 状态 1：未录制
- 主按钮：`开始诊断`

#### 状态 2：录制中
- 主按钮：`结束并提交`
- 次按钮：`查看会话详情`

#### 状态 3：上传失败
- 主按钮：`重试提交`

Popup 的控制要极简，不要塞太多分支动作。

---

### D. 会话摘要区
建议只展示最小摘要：
- 交互事件数
- 请求事件数
- 错误事件数
- 桥接事件数
- 已录制时长

作用：
- 给用户信心：系统确实在录

---

### E. 结果/跳转区
提交成功后展示：
- diagnosisId
- 生成状态
- `打开工作台`

可附一个次要入口：
- `复制诊断链接`

---

# 5. Popup 状态切换设计

这个很重要，建议明确成状态 UI。

---

## 5.1 Idle
```text
状态：未录制
按钮：[开始诊断]
说明：开始后将记录当前页面操作、请求与异常
```

---

## 5.2 Recording
```text
状态：录制中
信息：已录制 00:42
摘要：交互 8 / 请求 4 / 错误 1
按钮：[结束并提交] [查看会话详情]
```

---

## 5.3 Uploading
```text
状态：正在提交
提示：正在生成诊断会话，请稍候
按钮：禁用
```

---

## 5.4 Uploaded
```text
状态：提交成功
结果：诊断已创建
按钮：[打开工作台]
```

---

## 5.5 Failed
```text
状态：提交失败
提示：网络异常或服务暂不可用
按钮：[重试提交]
```

# 6. SidePanel 信息架构

如果 Popup 是控制台，SidePanel 就是会话详情辅助台。

---

## 6.1 SidePanel 页面结构
建议分 6 个区块：

1. 会话状态区
2. 页面上下文区
3. 事件时间线区
4. 用户备注区
5. 提交动作区
6. 结果跳转区

---

## 6.2 SidePanel 草图说明

```text
┌──────────────────────────────────┐
│ 当前诊断会话                      │
│ 状态：录制中                      │
│ 已录制：01:24                     │
├──────────────────────────────────┤
│ 页面上下文                        │
│ 应用：ERP Console                │
│ 模块：订单                        │
│ 页面：创建订单                    │
│ Route：/orders/create            │
│ Release：2026.06.16              │
├──────────────────────────────────┤
│ 事件时间线                        │
│ 10:01 点击 提交订单               │
│ 10:01 bridge: submitOrder        │
│ 10:01 POST /api/orders           │
│ 10:06 504 Gateway Timeout        │
│ 10:06 error toast 提交失败        │
├──────────────────────────────────┤
│ 用户补充                          │
│ 问题描述：__________________     │
│ 预期结果：__________________     │
│ 实际结果：__________________     │
├──────────────────────────────────┤
│ [结束并提交]                      │
│ [放弃本次会话]                    │
├──────────────────────────────────┤
│ 提交成功后：                      │
│ 诊断 #diag_xxx                   │
│ [打开工作台详情]                  │
└──────────────────────────────────┘
```

---

## 6.3 SidePanel 区块职责

### A. 会话状态区
展示：
- 状态
- 当前录制时长
- 关键计数摘要

作用：
- 比 Popup 更连续地反馈录制状态

---

### B. 页面上下文区
展示：
- appId
- module
- route/url
- releaseVersion
- tenant/project（如有）

作用：
- 帮助用户确认“这是哪一段场景”

---

### C. 事件时间线区
这是 SidePanel 的核心。

建议时间线以“用户可读摘要”为主，不展示太多原始 payload。  
优先展示：

- 用户交互
- bridge interaction
- network error
- ui symptom
- js error

不建议默认展示：
- 全量请求 headers
- 原始 stack 全文
- 大块 JSON

---

### D. 用户备注区
建议 3 个字段就够：

- 问题描述
- 预期结果
- 实际结果

为什么重要：
- 这是把“自动观测”补成“诊断输入”的关键一环
- 后端 explanation / AI 总结会更准

---

### E. 提交动作区
主要动作：

- `结束并提交`
- `放弃本次会话`

不要在这里放太多高级配置。

---

### F. 结果跳转区
提交成功后展示：
- diagnosisId
- 创建时间
- 打开工作台详情按钮

---

# 7. SidePanel 时间线设计建议

---

## 7.1 时间线节点类型
建议统一成 5 类：

1. **用户动作**
   - 点击按钮
   - 提交表单
   - 路由跳转

2. **业务桥接语义**
   - `submitOrder`
   - `loadList`
   - `retryPayment`

3. **网络事件**
   - 请求开始
   - 请求失败
   - 请求超时

4. **页面症状**
   - error toast
   - empty state
   - loading too long

5. **前端错误**
   - JS error
   - Promise rejection

---

## 7.2 展示原则
时间线展示要“用户能看懂”，建议一条记录只保留：

- 时间
- 类型标签
- 摘要文案

例如：

- `10:01 用户动作：点击“提交订单”`
- `10:01 业务动作：submitOrder`
- `10:01 网络请求：POST /api/orders`
- `10:06 网络异常：504 Gateway Timeout`
- `10:06 页面提示：提交失败`

这非常适合你后面映射成 domino chain。

# 8. 工作台入口信息架构

插件不是最终消费端，工作台才是。

---

## 8.1 插件到工作台的入口方式
建议插件只负责把用户送过去：

- `打开工作台详情`
- `复制诊断链接`

工作台 URL 建议可带：
- diagnosisId
- 可选 sessionId

---

## 8.2 工作台入口卡片
在插件里可以用一个轻量结果卡：

```text
诊断已生成
编号：diag_20260616_xxx
状态：成功
[打开工作台详情]
```

# 9. 页面之间的职责边界

这点一定要明确，不然后面页面会互相打架。

---

## Popup 负责什么
- 最短路径操作
- 开始/结束/提交
- 查看当前简要状态
- 跳工作台

## SidePanel 负责什么
- 录制中的会话详情
- 时间线预览
- 用户补充信息
- 提交前确认

## 工作台负责什么
- 正式诊断结果
- root cause
- supporting causes
- symptoms
- domino chain
- evidence drill-down

# 10. MVP 推荐最小页面集

如果你现在想更收口，我建议 MVP 只做：

## 必做
### Popup
- Idle
- Recording
- Uploading
- Uploaded
- Failed

### 工作台入口
- 打开结果页

## 可选但推荐
### SidePanel
- 时间线预览
- 用户备注
- 提交入口

如果资源紧，**第一版甚至可以先没有 SidePanel**，只做 Popup。  
因为最小闭环只靠 Popup 也能完成。
