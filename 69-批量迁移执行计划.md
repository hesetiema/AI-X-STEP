下面给你一个 **可直接执行的批量迁移清单**，目标是把 Diagnosis rules test 完整迁到你现在这套：

- `matcher`
- `factory`
- `rule helper assertions`
- `统一风格 rules unit tests`

我按你偏好的工程顺序来写：**先落文件结构，再替换 spec，再跑分层验证**。

### 原因
这样 rule spec 就不需要重复测 matcher 内部细节。

## Step 5：删除/收缩历史冗余 case

替换完 15 个 rule spec 后，清理以下旧风格内容：

### 删除这些类型的冗余断言
- 完整 finding object 深比较
- `title` / `summary` / `confidence` / `score` 精确断言
- `rule.name` / `rule.severity` 精确断言
- 同一 rule 下大量 matcher 细节 case
- rule-engine integration 中重复做单 rule unit 级断言

### 保留这些断言
- hit / miss
- `ruleCode`
- `layer`
- `cluster`
- `isSymptomOnly`
- `evidenceIds`
- 必要 `detail`
- 多入口规则补充 case（如 `R501`）

---

## Step 6：统一 import 路径和 helper 使用方式

所有 rule spec 尽量统一成这种结构：

```ts
import { XxxRule } from '../../../src/.../xxx.rule';
import { buildDiagnosisContext } from '../../helpers/diagnosis-context.fixture';
import {
  expectRuleMiss,
  expectSingleRuleHit,
} from '../../helpers/rule-unit-assertions';
```

这样后续新增 rule 的作者更容易复制已有样板。

---

## Step 7：最后再看 integration spec 是否需要微调

如果原来的 `rule-engine.service.int.spec.ts` 存在以下问题，再做轻微整理：

### 应保留
- 多条规则同时命中
- overlap recall
- dedupe key 行为
- symptom/root cause 共存
- 真实 ruleset 集成回归

### 应弱化/删除
- integration 中对单个 rule 的 full object 精确断言
- integration 中重复 matcher 级别判定细节

# 五、推荐失败排查顺序

如果某一轮挂了，不要直接全量看。按下面顺序排：

## 1. 先看 helper / import 路径
最常见问题：
- `buildDiagnosisContext` 路径不对
- `rule-unit-assertions.ts` 路径不对
- `RuleFinding` type import 路径不对

---

## 2. 再看 fixture shape 是否和真实 type 对齐
最常见问题：
- `evidence.type` 实际枚举名不同
- `symptoms` 实际结构不是 `string[]`
- `value` 内字段命名跟 matcher 实现不一致

---

## 3. 再看 matcher 阈值不一致
比如：
- timeout 阈值不是 3000
- loading 阈值不是 3000
- retry loop 条件不是你 spec 里假定的那样

这种情况优先改 spec 样例，不要先改 matcher，除非 matcher 行为本来就错。

---

## 4. 最后看 rule metadata 是否与你代码现状不同
比如：
- cluster 命名不一致
- layer 实际是 `frontend` 不是 `frontend_app`
- symptom 标记不同

如果是稳定业务语义不同，再改 spec。

# 八、最终验收标准

迁移完成后，你应该达到：

## 结构上
- 所有 rule spec 命名统一
- 所有 rule spec 使用统一 helper
- matcher / factory / rules / integration 职责分层清晰

## 测试上
- `test/unit/rules` 全绿
- `test/unit/rule-engine/matchers` 全绿
- `test/unit/rule-engine/factories` 全绿
- `test/integration/rule-engine.service.int.spec.ts` 全绿

## 维护上
新增一条 rule 时，作者只需要：
1. 写 matcher（可选）
2. 写 factory 风格 rule
3. 复制一个统一 spec 模板
