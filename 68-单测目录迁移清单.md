# 五、哪些旧 spec 可以删除/替换

下面按“迁移规则”给你，最实用。

---

## 1. 旧的 Rule 单测：**保留文件名，替换内容**
如果你现在已经有这些文件：

- `r201-api-5xx.rule.spec.ts`
- `r202-api-timeout.rule.spec.ts`
- `r203-bff-gateway-failure.rule.spec.ts`
- ...
- `r504-ui-retry-loop-symptom.rule.spec.ts`

### 建议：
- **不要新建第二份**
- 直接**覆盖为统一 helper 风格版**

### 原因：
- 避免同一规则出现两套 spec
- 避免 Jest 重复执行近义测试
- 保留 git history 更清晰

即：

| 旧文件存在？ | 动作 |
|---|---|
| 是 | 直接替换内容 |
| 否 | 新建对应 spec |

---

## 2. 旧的 Rule 单测如果是“超细节断言版”：建议删除冗余 case
例如旧 spec 里如果大量断言这些：

- `title`
- `summary`
- `confidence`
- `score`
- `rule.name`
- `rule.severity`
- 完整 `detail` 深比较
- 完整 finding object 深比较

### 建议：
删除这些冗余 case，保留/改写成统一风格：

- hit
- miss
- 必要 detail
- 多入口规则的额外 case（如 `R501`）

### 原因：
这些细节现在已经分别由：
- matcher tests
- factory tests
承担了更稳定的覆盖职责。

---

## 3. 旧的 matcher 前时代 spec：可以删除“重复验证 matcher 内部逻辑”的 rule case
比如某条 rule spec 里如果写了很多这种 case：

- timeout by label
- timeout by code
- timeout by duration threshold
- gateway by status 502
- gateway by label contains gateway
- empty state by expectedCount > 0
- retry loop by retryCount >= 3

这些本质上是 **matcher spec 的职责**。

### 建议：
从 rule spec 里删掉，留在 matcher spec 中。

### Rule spec 只保留：
- 一条最小命中样例
- 一条最小未命中样例
- 若是多入口规则，再加一条 alternate path 命中样例

---

## 4. 旧的 “exact full object snapshot” rule spec：建议删除
如果你有类似：

```ts
expect(rule.evaluate(context)).toEqual([
  {
    ruleCode: 'R201',
    title: 'API 5xx root cause',
    summary: '...',
    confidence: 0.91,
    score: 91,
    ...
  },
]);
```

### 建议：
删除或改成 `expectSingleRuleHit(...)`

### 原因：
这种 spec 在你调文案/分值时会非常脆。

---

## 5. 旧的 RuleEngineService 粒度测试里如果单测某条规则：删除重复覆盖
例如某些历史 integration spec 可能既测试了引擎又顺手深测单条 rule：

- “service.evaluate(context) should return exactly R201 with full object”
- “service.evaluate(context) should return exactly R202 ...”

### 建议：
如果只是重复单条 rule 的命中行为，删掉或弱化。

### integration 层保留：
- 多规则共存
- overlapping hit
- dedupe
- scenario-level recall

# 九、一个很实用的迁移判断标准

如果你在犹豫某个旧 spec 要不要删，就用这个规则：

## 保留它，只有当它回答了下面至少一个问题：
1. 这条规则是否应该 hit / miss？
2. 这条规则是否是 symptom / root cause？
3. 这条规则的 layer / cluster 是否正确？
4. 多条规则是否可以共存召回？
5. dedupe 是否正确？
6. finding factory 是否正确构造标准输出？
7. matcher 是否正确识别底层 evidence 模式？

如果一个旧 spec **没有新增回答这些问题**，只是重复另一层已经验证过的事实，就删。
