# 1. Diagnosis Service 的职责边界

先把边界定清楚，不然后面容易把 Rule Engine 和 Diagnosis Service 混在一起。

---

## 1.1 Rule Engine 做什么

Rule Engine 负责：

- 根据证据判断哪些规则命中
- 输出多个 `RuleFinding`
- 给每个 finding 一个基础置信度

例如输出：

- `R101 STORE_NOT_UPDATED`
- `R201 FALSY_VALUE_SWALLOWED`
- `R403 REQUEST_SUCCESS_BUT_PAGE_NO_CHANGE`

---

## 1.2 Diagnosis Service 做什么

Diagnosis Service 负责：

- 对多个 findings 做 **归因排序**
- 选择 **最可能根因**
- 合并关联规则
- 生成最终诊断结论
- 输出给工作台/诊断报告

也就是：

> **Rule Engine 产出候选原因，Diagnosis Service 决定最终怎么讲。**

---

# 2. 归因排序要解决什么问题

同一个问题往往会命中多条规则。

比如一个场景：

- 请求成功
- store 没更新
- 页面也没变化

可能会同时命中：

- `R101 STORE_NOT_UPDATED`
- `R403 REQUEST_SUCCESS_BUT_PAGE_NO_CHANGE`

但对用户来说，这两个不是同层信息：

- `R101` 更像根因
- `R403` 更像结果表现

所以 Diagnosis Service 要解决 4 个问题：

---

## 2.1 根因优先于表现

优先输出“为什么会这样”，不是“现象是什么”。

例如：

- 优先：`STORE_NOT_UPDATED`
- 次级：`REQUEST_SUCCESS_BUT_PAGE_NO_CHANGE`

---

## 2.2 上游优先于下游

同一条链路中，上游问题通常更接近真正原因。

链路：

```text
API → Store → Selector → Render → DOM
```

排序优先级应该倾向：

```text
API > Store > Selector > Render > DOM
```

---

## 2.3 具体规则优先于泛规则

例如：

- `R201 FALSY_VALUE_SWALLOWED`
- `R202 FORMATTER_OUTPUT_FALLBACK`

如果都命中，应该优先 `R201`，因为它更具体、更可修复。

---

## 2.4 证据完整度高的优先

如果某条规则虽然理论上命中，但证据链很弱，不应该排第一。

例如：

- `R302 DOM_HIDDEN_OR_INVISIBLE` 只有 DOM snapshot，没有 computed style
- `R101 STORE_NOT_UPDATED` 有 response / store binding / state timeline 全证据

那更应该优先 `R101`。

---

# 3. 排序总体策略

我建议 Diagnosis Service 用一个 **综合得分模型**，而不是只看单个 confidence。

---

## 3.1 最终排序分 = 规则基础分 + 归因修正分

```text
final_score =
  confidence_score
+ root_cause_bonus
+ upstream_bonus
+ specificity_bonus
+ evidence_completeness_bonus
+ chain_consistency_bonus
- symptom_penalty
- duplicate_penalty
```

---

## 3.2 每部分含义

### 1）confidence_score
来自 Rule Engine 的基础置信度。

### 2）root_cause_bonus
如果是明显根因类规则，加分。

### 3）upstream_bonus
更靠上游层级，加分。

### 4）specificity_bonus
更具体、解释力更强的规则，加分。

### 5）evidence_completeness_bonus
证据越完整，加分越高。

### 6）chain_consistency_bonus
如果和整条链路观测结果一致，再加分。

### 7）symptom_penalty
纯表现型规则，扣分。

### 8）duplicate_penalty
和更强规则重复表达同一问题，扣分。

---

# 4. 建议的排序维度

我建议 MVP 用 6 个核心维度就够了。

---

## 4.1 维度 A：基础置信度

直接来自 rule finding：

```ts
finding.confidence // 0 ~ 1
```

映射成 0~100：

```ts
baseScore = finding.confidence * 100
```

---

## 4.2 维度 B：层级根因权重

建议给每个 layer 一个权重：

```ts
const LAYER_ROOT_CAUSE_WEIGHT: Record<Layer, number> = {
  api: 30,
  api_to_state: 26,
  state: 24,
  selector: 20,
  state_to_render: 18,
  render: 16,
  dom: 10,
  handler: 14,
  ui: 8,
  request_to_ui: 12,
};
```

解释：
- `api/state` 更接近根因
- `dom/ui` 更偏表象

---

## 4.3 维度 C：规则具体性权重

```ts
const RULE_SPECIFICITY_WEIGHT: Record<string, number> = {
  R201: 20,
  R001: 18,
  R101: 18,
  R102: 16,
  R203: 15,
  R403: 10,
  R202: 8,
  R301: 7,
  R302: 7,
  R303: 6,
  R003: 10,
  R002: 9,
  R103: 12,
  R402: 10,
  R401: 9,
  R204: 6,
};
```

---

## 4.4 维度 D：证据完整度

证据完整度建议按“期望证据数 / 实际命中证据数”来估算。

例如：

```ts
const EXPECTED_EVIDENCE_COUNT: Record<string, number> = {
  R001: 3,
  R101: 3,
  R201: 3,
  R202: 2,
  R203: 2,
  R301: 3,
  R302: 3,
  R403: 4,
};
```

计算：

```ts
evidenceCompleteness = actualEvidenceCount / expectedEvidenceCount
```

再乘一个系数：

```ts
evidenceBonus = evidenceCompleteness * 15
```

---

## 4.5 维度 E：链路一致性

如果规则和当前模式/链路非常一致，要加分。

例如：

### inspect 模式下
`API → Store → Render → DOM`

- `R001/R101/R201/R203/R301` 更一致

### click 模式下
`Click → Handler → Request → Response → State → Render → DOM`

- `R401/R402/R403` 更一致

可做简单加分：

```ts
chainConsistencyBonus = 0 ~ 10
```

---

## 4.6 维度 F：症状扣分

有些规则更多是在描述现象，而不是根因：

- `R403 REQUEST_SUCCESS_BUT_PAGE_NO_CHANGE`
- `R301 DOM_NOT_UPDATED`
- `R303 DOM_OVERWRITTEN_BY_FALLBACK`

这些可以有一个 symptom penalty：

```ts
const SYMPTOM_PENALTY_RULES = {
  R403: 8,
  R301: 6,
  R303: 6,
};
```

## 6.2 ranking.ts

```ts
import { DiagnosisContext, RuleFinding } from './types';
import {
  EXPECTED_EVIDENCE_COUNT,
  LAYER_ROOT_CAUSE_WEIGHT,
  MODE_CHAIN_BONUS,
  RULE_SPECIFICITY_WEIGHT,
  SYMPTOM_PENALTY_RULES,
} from './constants';

export interface DiagnosisRankScore {
  ruleCode: string;
  baseScore: number;
  rootCauseScore: number;
  specificityScore: number;
  evidenceScore: number;
  chainConsistencyScore: number;
  symptomPenalty: number;
  duplicatePenalty: number;
  finalScore: number;
}

export interface EnrichedFinding extends RuleFinding {
  rankScore: DiagnosisRankScore;
  isRootCauseCandidate: boolean;
  isSymptomOnly: boolean;
  relatedFindings: string[];
  diagnosisLabel: string;
}

function calcBaseScore(finding: RuleFinding): number {
  return finding.confidence * 100;
}

function calcRootCauseScore(finding: RuleFinding): number {
  return LAYER_ROOT_CAUSE_WEIGHT[finding.layer] ?? 0;
}

function calcSpecificityScore(finding: RuleFinding): number {
  return RULE_SPECIFICITY_WEIGHT[finding.ruleCode] ?? 0;
}

function calcEvidenceScore(finding: RuleFinding): number {
  const expected = EXPECTED_EVIDENCE_COUNT[finding.ruleCode] ?? 1;
  const actual = finding.evidenceRefs?.length ?? 0;
  const completeness = Math.min(actual / expected, 1);

  return completeness * 15;
}

function calcChainConsistencyScore(
  finding: RuleFinding,
  ctx: DiagnosisContext
): number {
  return MODE_CHAIN_BONUS[ctx.mode]?.[finding.ruleCode] ?? 0;
}

function calcSymptomPenalty(finding: RuleFinding): number {
  return SYMPTOM_PENALTY_RULES[finding.ruleCode] ?? 0;
}

function calcDuplicatePenalty(
  finding: RuleFinding,
  allFindings: RuleFinding[]
): number {
  const sameClusterCount = allFindings.filter(
    (f) => f.cluster && finding.cluster && f.cluster === finding.cluster
  ).length;

  if (sameClusterCount <= 1) return 0;

  // 同簇多个规则同时命中，弱规则要降权
  return (sameClusterCount - 1) * 4;
}

function buildDiagnosisLabel(finding: RuleFinding): string {
  switch (finding.ruleCode) {
    case 'R001':
      return '接口字段缺失';
    case 'R101':
      return '状态未更新';
    case 'R201':
      return '合法值被判空吞掉';
    case 'R202':
      return '格式化输出兜底值';
    case 'R203':
      return '状态变化未触发渲染';
    case 'R301':
      return '渲染结果未落到 DOM';
    case 'R403':
      return '请求成功但页面无变化';
    default:
      return finding.title;
  }
}

function isRootCauseCandidate(finding: RuleFinding): boolean {
  return ['api', 'api_to_state', 'state', 'selector', 'state_to_render', 'render'].includes(
    finding.layer
  );
}

function isSymptomOnly(finding: RuleFinding): boolean {
  return ['R301', 'R303', 'R403'].includes(finding.ruleCode);
}

export function scoreFinding(
  finding: RuleFinding,
  allFindings: RuleFinding[],
  ctx: DiagnosisContext
): DiagnosisRankScore {
  const baseScore = calcBaseScore(finding);
  const rootCauseScore = calcRootCauseScore(finding);
  const specificityScore = calcSpecificityScore(finding);
  const evidenceScore = calcEvidenceScore(finding);
  const chainConsistencyScore = calcChainConsistencyScore(finding, ctx);
  const symptomPenalty = calcSymptomPenalty(finding);
  const duplicatePenalty = calcDuplicatePenalty(finding, allFindings);

  const finalScore =
    baseScore +
    rootCauseScore +
    specificityScore +
    evidenceScore +
    chainConsistencyScore -
    symptomPenalty -
    duplicatePenalty;

  return {
    ruleCode: finding.ruleCode,
    baseScore,
    rootCauseScore,
    specificityScore,
    evidenceScore,
    chainConsistencyScore,
    symptomPenalty,
    duplicatePenalty,
    finalScore,
  };
}

export function rankFindings(
  findings: RuleFinding[],
  ctx: DiagnosisContext
): EnrichedFinding[] {
  const enriched: EnrichedFinding[] = findings.map((finding) => ({
    ...finding,
    rankScore: scoreFinding(finding, findings, ctx),
    isRootCauseCandidate: isRootCauseCandidate(finding),
    isSymptomOnly: isSymptomOnly(finding),
    relatedFindings: findings
      .filter((f) => f.ruleCode !== finding.ruleCode && f.cluster === finding.cluster)
      .map((f) => f.ruleCode),
    diagnosisLabel: buildDiagnosisLabel(finding),
  }));

  return enriched.sort((a, b) => {
    if (b.rankScore.finalScore !== a.rankScore.finalScore) {
      return b.rankScore.finalScore - a.rankScore.finalScore;
    }

    if (b.confidence !== a.confidence) {
      return b.confidence - a.confidence;
    }

    return a.ruleCode.localeCompare(b.ruleCode);
  });
}
```

# 7. 归因合并策略

排序之后，不建议直接把所有 findings 原样返回。  
Diagnosis Service 还应该做一次 **归因合并**。

---

## 7.1 合并目标

比如同时命中：

- `R201 FALSY_VALUE_SWALLOWED`
- `R202 FORMATTER_OUTPUT_FALLBACK`

最终不一定都作为“主结论”展示。

而是：

- 主结论：`R201 合法值被判空吞掉`
- 辅助说明：`R202 格式化函数输出了 fallback`

---

## 7.2 合并规则建议

### 规则 A：同 cluster 合并
同一 cluster 只保留一个主 finding，其余作为 support finding。

### 规则 B：根因 finding 主展示
根因 finding 放主卡片。

### 规则 C：表现型 finding 下沉
表现型 findings 放到“影响表现”或“关联证据”。

---

## 7.3 合并后输出结构

```ts
export interface DiagnosisConclusion {
  topFinding: EnrichedFinding | null;
  supportingFindings: EnrichedFinding[];
  symptomFindings: EnrichedFinding[];
  summary: string;
  repairHints: string[];
}
```

---

## 7.4 合并代码骨架

```ts
export function buildDiagnosisConclusion(
  ranked: EnrichedFinding[]
): DiagnosisConclusion {
  if (!ranked.length) {
    return {
      topFinding: null,
      supportingFindings: [],
      symptomFindings: [],
      summary: '当前证据不足，未能形成明确归因结论。',
      repairHints: [],
    };
  }

  const topFinding = ranked[0];

  const supportingFindings = ranked.filter(
    (f, index) =>
      index > 0 &&
      !f.isSymptomOnly &&
      f.cluster !== topFinding.cluster
  );

  const symptomFindings = ranked.filter((f) => f.isSymptomOnly);

  const repairHints = Array.from(
    new Set([
      ...topFinding.suggestions,
      ...supportingFindings.flatMap((f) => f.suggestions).slice(0, 3),
    ])
  ).slice(0, 5);

  return {
    topFinding,
    supportingFindings,
    symptomFindings,
    summary: `最可能根因是「${topFinding.diagnosisLabel}」：${topFinding.summary}`,
    repairHints,
  };
}
```

# 8. 典型场景排序示例

---

## 场景 1：请求成功，但 store 没更新，页面也没变化

命中：

- `R101` confidence=0.90
- `R403` confidence=0.93

### 为什么 `R101` 应排前面？

虽然 `R403` confidence 更高，但它更偏“结果描述”  
`R101` 更接近真正中断点。

### 预期排序

```text
1. R101 STORE_NOT_UPDATED
2. R403 REQUEST_SUCCESS_BUT_PAGE_NO_CHANGE
```

---

## 场景 2：上游值是 0，formatter 输出 '--'

命中：

- `R201` confidence=0.94
- `R202` confidence=0.86
- `R303` confidence=0.82

### 预期排序

```text
1. R201 FALSY_VALUE_SWALLOWED
2. R202 FORMATTER_OUTPUT_FALLBACK
3. R303 DOM_OVERWRITTEN_BY_FALLBACK
```

解释：
- `R201` 最具体、最接近根因
- `R202` 是直接技术表现
- `R303` 是最终页面表现

---

## 场景 3：store 更新了，但组件没 render，页面没变

命中：

- `R102`
- `R203`
- `R403`

### 预期排序

```text
1. R102 STORE_UPDATED_BUT_NOT_SUBSCRIBED
2. R203 RENDER_NOT_TRIGGERED
3. R403 REQUEST_SUCCESS_BUT_PAGE_NO_CHANGE
```

解释：
- `R102` 比 `R203` 更具体
- `R403` 只是结果

# 9. 最终 Diagnosis Service 建议输出

我建议给前端工作台的最终输出，不要直接是“排序结果数组”，而是一个更接近产品消费的结构：

```ts
export interface DiagnosisServiceResponse {
  diagnosisId: string;
  interactionId?: string;
  lineageId?: string;

  topCause: EnrichedFinding | null;
  supportingCauses: EnrichedFinding[];
  symptoms: EnrichedFinding[];

  summary: string;
  repairHints: string[];

  scoreBreakdown?: Array<{
    ruleCode: string;
    finalScore: number;
    reason: string[];
  }>;
}
```

# 10. 排序理由可解释化

为了让工作台更有说服力，建议把排序理由也输出。

比如：

```ts
{
  ruleCode: 'R101',
  finalScore: 156,
  reason: [
    '属于状态层根因，优先级高',
    '证据完整度高（3/3）',
    '与当前 click_diagnosis 链路一致',
    '比 R403 更接近真正中断点'
  ]
}
```

这个对你后面做“多米诺骨牌式因果链解释”特别有用。

# 11. MVP 建议的最小实现

第一版 Diagnosis Service 不要太复杂，我建议只做：

### 必做
- 综合打分
- 排序
- top1 选取
- supporting / symptom 分组
- summary 生成

### 暂缓
- ML 排序
- 历史案例学习
- 用户反馈回灌
- 规则动态权重学习

# 12. 一句话总结

Diagnosis Service 的核心不是“再判断一次规则”，而是：

> **把 Rule Engine 产出的多个候选原因，按照“根因优先、上游优先、具体优先、证据完整优先”的原则，整理成用户真正能消费的诊断结论。**
