下面直接给你 **Rule 文件拆分后的 Jest unit test 模板全集**。  
我按你前面已经采用的 skeleton 风格来写，特点是：

- **每条规则单文件单测**
- 断言尽量聚焦：
  - hit / not hit
  - `ruleCode`
  - `layer`
  - `cluster`
  - `isSymptomOnly`
  - `evidenceIds`
- 避免过度绑定实现细节，便于后续规则召回增强时少改测试

import { R101UserActionMissingRule } from '../../../src/diagnosis/domain/rule-engine/rules/r101-user-action-missing.rule';
import { buildDiagnosisContext } from '../../helpers/diagnosis-context.fixture';

describe('R101UserActionMissingRule', () => {
  let rule: R101UserActionMissingRule;

  beforeEach(() => {
    rule = new R101UserActionMissingRule();
  });

  it('should hit when no user action evidence exists for the interaction', () => {
    const context = buildDiagnosisContext({
      evidence: [
        {
          id: 'ev-001',
          type: 'ui_state',
          label: 'loading > 10s',
          value: { loadingMs: 10000 },
          source: 'frontend-sdk',
        },
      ],
    });

    const result = rule.evaluate(context);

    expect(result).toHaveLength(1);
    expect(result[0].ruleCode).toBe('R101');
    expect(result[0].layer).toBe('user_action');
    expect(result[0].cluster).toBe('missing_user_action');
    expect(result[0].isSymptomOnly).toBe(false);
  });

  it('should not hit when user action evidence is present', () => {
    const context = buildDiagnosisContext({
      evidence: [
        {
          id: 'ev-action-1',
          type: 'ui_event',
          label: 'click submit',
          value: { element: '#submit-btn' },
          source: 'frontend-sdk',
        },
      ],
    });

    const result = rule.evaluate(context);

    expect(result).toEqual([]);
  });
});

# 八、推荐一个批量断言策略

你后面规则越来越多时，建议每个规则单测统一维持这两类 case：

## 1）正向命中
断言：

```ts
expect(result).toHaveLength(1);
expect(result[0].ruleCode).toBe('Rxxx');
expect(result[0].layer).toBe('...');
expect(result[0].cluster).toBe('...');
```

## 2）负向不命中
断言：

```ts
expect(result).toEqual([]);
```

不要一开始就把：
- `title`
- `summary`
- `confidence`
- `detail`
- `score`

全部写死，否则你后面调规则文案或置信度时，测试会很脆。
