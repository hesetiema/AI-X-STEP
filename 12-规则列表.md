下面直接给你一版 **Rule Engine 规则清单 v0.1**。  
目标不是一开始做“万能规则系统”，而是先覆盖 MVP 最常见、最有价值、最容易解释的异常类型。

我会按这几个部分来给：

1. Rule Engine 在 MVP 里的定位  
2. 规则分层  
3. 规则清单 v0.1  
4. 每条规则的触发条件 / 输入证据 / 输出格式  
5. 规则优先级建议  
6. MVP 先实现哪几条

# 1. Rule Engine 在 MVP 里的定位

Rule Engine 不负责“采集事实”，也不负责“最终自然语言解释”。

它只做一件事：

> **根据 Trace + Lineage + Evidence，对问题进行确定性归因。**

所以它输入的是结构化证据：

- trace nodes
- causality edges
- lineage steps
- evidence summaries
- metadata bindings

输出的是结构化 finding：

- 命中哪条规则
- 置信度
- 为什么命中
- 对应哪一层
- 建议怎么修

---

# 2. 规则分层

为了便于实现和扩展，我建议 v0.1 把规则分 5 类：

```text
R1. 数据源层规则
R2. 状态/绑定层规则
R3. 渲染/转换层规则
R4. DOM展示层规则
R5. 交互链路层规则
```

对应你关心的链路：

```text
API → Store → Render → DOM
以及
Click → Request → Response → State → Render → DOM
```

# 3. Rule Engine 规则清单 v0.1

下面是我建议的第一批规则，共 12 条。  
已经足够支撑 MVP。

---

## A. 数据源层规则

### R001. API_FIELD_MISSING
**接口字段缺失**

#### 典型场景
页面需要 `response.data.amount`，但接口里没有这个字段。

#### 用户感知
- 页面显示空
- 页面显示 `--`
- 页面显示默认值

---

### R002. API_VALUE_NULL
**接口字段存在但值为 null / undefined**

#### 典型场景
接口有字段，但值为空，页面 fallback。

---

### R003. RESPONSE_NOT_CONSUMED
**接口成功返回，但前端未消费**

#### 典型场景
请求成功了，response 里有数据，但没有进入 store / render 链路。

---

## B. 状态/绑定层规则

### R101. STORE_NOT_UPDATED
**请求成功后 store 未更新**

#### 典型场景
response 回来了，但 reducer / setter 没执行或没生效。

---

### R102. STORE_UPDATED_BUT_NOT_SUBSCRIBED
**store 已更新，但组件未订阅/未感知**

#### 典型场景
数据变了，但组件没有 re-render。

---

### R103. SELECTOR_RETURNS_EMPTY
**selector / 数据映射结果为空**

#### 典型场景
store 里有值，但 selector 取错路径、过滤后为空。

---

## C. 渲染/转换层规则

### R201. FALSY_VALUE_SWALLOWED
**0 / false / '' 被错误判空吞掉**

#### 典型场景
`if (!value)` 把 `0` 当成空处理。

---

### R202. FORMATTER_OUTPUT_FALLBACK
**formatter 输出 fallback 值**

#### 典型场景
前端格式化函数输出 `--`、`暂无`、`N/A`。

---

### R203. RENDER_NOT_TRIGGERED
**数据变化后未触发 render**

#### 典型场景
state 已变，但 render 节点没出现。

---

### R204. RENDER_OUTPUT_UNCHANGED
**render 发生了，但输出未变化**

#### 典型场景
组件刷新了，但最终渲染结果没变，可能是表达式逻辑有问题。

---

## D. DOM 展示层规则

### R301. DOM_NOT_UPDATED
**render 发生了，但 DOM 未更新**

#### 典型场景
virtual dom 有变化，但最终页面没变化，或目标节点没被更新。

---

### R302. DOM_HIDDEN_OR_INVISIBLE
**DOM 存在但不可见**

#### 典型场景
有节点，但被 `display:none` / `visibility:hidden` / 遮挡 / 折叠。

---

### R303. DOM_OVERWRITTEN_BY_FALLBACK
**目标 DOM 被兜底文案覆盖**

#### 典型场景
真实值存在，但页面逻辑最终显示统一 fallback 文案。

---

## E. 交互链路层规则

### R401. CLICK_NO_HANDLER
**点击后无 handler**

#### 典型场景
按钮点了，但没有绑定处理函数，或埋点未捕获到 handler。

---

### R402. HANDLER_NO_REQUEST_NO_STATE_CHANGE
**点击后既没请求也没状态变化**

#### 典型场景
点击事件执行了，但后续什么都没发生。

---

### R403. REQUEST_SUCCESS_BUT_PAGE_NO_CHANGE
**请求成功，但页面没有可观察变化**

#### 典型场景
response OK，但没有 state/render/dom 的后续链路。

---

# 5. 逐条规则说明

下面我给你一版更工程化的规则定义。

---

## R001 API_FIELD_MISSING

### 触发条件
- lineage 目标路径中存在目标 api field
- network response 成功
- response body 中缺失该字段
- 下游 DOM 显示 fallback/空值

### 需要证据
- response snapshot
- metadata 中 component → api_field 绑定
- dom displayed value

### 输出
- title: 接口缺少页面所需字段
- severity: high

### 建议
- 检查接口契约
- 检查字段兼容性
- 增加 response schema 校验

---

## R002 API_VALUE_NULL

### 触发条件
- response body 中字段存在
- 值为 `null` / `undefined`
- 页面显示 fallback

### 说明
这和字段缺失要分开，因为修复责任可能不同。

---

## R003 RESPONSE_NOT_CONSUMED

### 触发条件
- network_response 成功
- response 中存在目标字段和值
- 没有对应 state_update / render 消费证据

### 建议
- 检查 response 到 store 的映射逻辑
- 检查 promise/async 回调链

---

## R101 STORE_NOT_UPDATED

### 触发条件
- response 成功
- 预期 store key 已知
- 在时间窗口内没有该 store key 的更新

### 证据
- response snapshot
- state timeline
- metadata: api_field -> store_key

---

## R102 STORE_UPDATED_BUT_NOT_SUBSCRIBED

### 触发条件
- store key 已更新
- 目标组件未发生 render
- 或 render 与该 store key 无关联消费

### 典型根因
- selector 没订阅到
- 组件 memo 过度
- 依赖数组错误
- props 未透传

---

## R103 SELECTOR_RETURNS_EMPTY

### 触发条件
- store 中目标值存在
- selector / mapper 输出为空
- render 层拿到的是空值

### 说明
这个规则很适合排查“store 里明明有，但页面没显示”。

---

## R201 FALSY_VALUE_SWALLOWED

### 触发条件
- api/store value 为 `0` / `false` / `""`
- formatter/render expr 输出 fallback token
- fallback token 命中白名单：`--`, `暂无`, `N/A`, `-`

### 置信度建议
高

### 这是 MVP 第一优先级规则之一

---

## R202 FORMATTER_OUTPUT_FALLBACK

### 触发条件
- formatter 节点存在
- formatter 输入非空
- formatter 输出 fallback token

### 与 R201 的关系
R202 更泛，R201 是其特例。

---

## R203 RENDER_NOT_TRIGGERED

### 触发条件
- store 或 props 变化
- 在合理时间窗口内无 render_commit 节点

### 适合定位
- 组件未刷新
- 订阅没生效
- diff 被跳过

---

## R204 RENDER_OUTPUT_UNCHANGED

### 触发条件
- render_commit 发生
- render output snapshot 与前一次一致
- 但上游 state/props 已变化

### 说明
这条规则适合发现“刷新了但还是旧值”。

---

## R301 DOM_NOT_UPDATED

### 触发条件
- render output 发生变化
- 目标 DOM 节点文本/属性未变化

### 典型问题
- key 不稳定
- 节点命中错误
- 条件分支没进入目标节点

---

## R302 DOM_HIDDEN_OR_INVISIBLE

### 触发条件
- DOM 节点存在
- 目标值存在
- 但 visibility/display/opacity/尺寸等导致不可见

### 证据
- DOM snapshot
- computed style
- bounding box

---

## R303 DOM_OVERWRITTEN_BY_FALLBACK

### 触发条件
- render 或 state 中存在真实值
- DOM 最终显示 fallback 文案
- 存在统一兜底模板/placeholder 逻辑

### 和 R202 区别
R202 更偏 formatter  
R303 更偏最终模板覆盖

---

## R401 CLICK_NO_HANDLER

### 触发条件
- click_event 存在
- 无 handler_start

### 适合发现
- disabled 按钮
- 事件未绑定
- 冒泡/遮罩拦截

---

## R402 HANDLER_NO_REQUEST_NO_STATE_CHANGE

### 触发条件
- handler_start 存在
- 无 network_request
- 无 state_update
- 无后续 dom change

### 说明
适合定位“点了没反应”。

---

## R403 REQUEST_SUCCESS_BUT_PAGE_NO_CHANGE

### 触发条件
- network_response 成功
- 后续无 state/render/dom 关键节点
- 用户界面未观察到变化

### 这是点击链路诊断里很关键的一条

# 7. 规则优先级建议

为了 MVP 尽快落地，我建议分 3 批。

---

## P0：第一批必须做
这些最容易出效果，且命中频率高。

1. `R201 FALSY_VALUE_SWALLOWED`
2. `R202 FORMATTER_OUTPUT_FALLBACK`
3. `R101 STORE_NOT_UPDATED`
4. `R203 RENDER_NOT_TRIGGERED`
5. `R403 REQUEST_SUCCESS_BUT_PAGE_NO_CHANGE`
6. `R001 API_FIELD_MISSING`

---

## P1：第二批建议做
1. `R102 STORE_UPDATED_BUT_NOT_SUBSCRIBED`
2. `R103 SELECTOR_RETURNS_EMPTY`
3. `R301 DOM_NOT_UPDATED`
4. `R302 DOM_HIDDEN_OR_INVISIBLE`
5. `R402 HANDLER_NO_REQUEST_NO_STATE_CHANGE`

---

## P2：第三批增强
1. `R002 API_VALUE_NULL`
2. `R003 RESPONSE_NOT_CONSUMED`
3. `R204 RENDER_OUTPUT_UNCHANGED`
4. `R303 DOM_OVERWRITTEN_BY_FALLBACK`
5. `R401 CLICK_NO_HANDLER`

# 8. MVP 第一版推荐最小规则集

如果你真的要非常克制，我建议第一版只做 **6 条**：

```text
R001 API_FIELD_MISSING
R101 STORE_NOT_UPDATED
R201 FALSY_VALUE_SWALLOWED
R202 FORMATTER_OUTPUT_FALLBACK
R203 RENDER_NOT_TRIGGERED
R403 REQUEST_SUCCESS_BUT_PAGE_NO_CHANGE
```

这 6 条已经能覆盖：

- 接口没给
- response 没入状态
- 状态变化没触发渲染
- formatter 把值吞了
- 请求成功但页面没反应

基本就是 MVP 最常见的 80% 场景。

---

# 9. 建议的规则执行顺序

不要所有规则乱跑，建议按层次顺序执行：

```text
第一层：数据源层
R001 / R002 / R003

第二层：状态绑定层
R101 / R102 / R103

第三层：渲染转换层
R201 / R202 / R203 / R204

第四层：DOM层
R301 / R302 / R303

第五层：交互链路层
R401 / R402 / R403
```

好处是：
- 更符合人的理解顺序
- 更适合生成解释
- 更容易做置信度收敛

# 10. 规则冲突处理建议

有些规则会同时命中，比如：

- `R202 FORMATTER_OUTPUT_FALLBACK`
- `R201 FALSY_VALUE_SWALLOWED`

这时建议：

## 原则 1：具体规则优先于泛规则
- `R201` 比 `R202` 更具体
- 报告中优先把 `R201` 放前面

## 原则 2：上游根因优先于下游表现
- `API_FIELD_MISSING` 应优先于 `DOM_OVERWRITTEN_BY_FALLBACK`

## 原则 3：根因类优先于结果类
- `STORE_NOT_UPDATED` 优先于 `REQUEST_SUCCESS_BUT_PAGE_NO_CHANGE`

# 11. 规则配置表示例

后面可以做成配置文件驱动。先给你个 YAML 风格样例：

```yaml
- rule_code: R201
  name: FALSY_VALUE_SWALLOWED
  category: render_transform
  severity: high
  priority: 100
  fallback_tokens: ["--", "-", "暂无", "N/A"]
  conditions:
    - upstream_value_in: [0, false, ""]
    - formatter_output_in_tokens: true
  suggestions:
    - "将 if (!value) 修改为 value == null"
    - "补充 0/false 场景测试"

- rule_code: R101
  name: STORE_NOT_UPDATED
  category: state_binding
  severity: high
  priority: 90
  conditions:
    - response_success: true
    - expected_store_key_exists: true
    - no_store_update_in_window_ms: 1000
  suggestions:
    - "检查 response 到 store 的映射逻辑"
    - "检查 reducer/setter 是否执行"
```

# 12. 一句话总结

这版 **Rule Engine 规则清单 v0.1** 的核心思想是：

> 先围绕 **API → Store → Render → DOM** 和 **Click → Request → Response → State → Render → DOM**  
> 做一批**可解释、可验证、可展示**的规则。

MVP 阶段最重要的不是规则多，而是：

- 命中后用户能看懂
- 工程上容易验证
- 能稳定支撑诊断报告
