# 1. 判定引擎整体流程

Rule Engine 的执行建议不要“所有规则平铺跑一遍”，而是按层执行：

```text
Step 1. 归一化证据
Step 2. 建立诊断上下文 DiagnosisContext
Step 3. 按层执行规则
        - 数据源层
        - 状态层
        - 渲染层
        - DOM 层
        - 交互链路层
Step 4. 对命中结果打分
Step 5. 去重 / 合并 / 排序
Step 6. 输出 findings
```

---

## 1.1 总伪代码

```python
def run_rule_engine(context: DiagnosisContext) -> list[RuleFinding]:
    findings = []

    normalize_context(context)

    findings += run_data_source_rules(context)
    findings += run_state_binding_rules(context)
    findings += run_render_rules(context)
    findings += run_dom_rules(context)
    findings += run_interaction_rules(context)

    findings = deduplicate_findings(findings)
    findings = merge_related_findings(findings)
    findings = rerank_findings(findings, context)

    return findings
```

## 2.1 DiagnosisContext

```python
class DiagnosisContext:
```

## R001 API_FIELD_MISSING

### 判定逻辑
如果：
- response 成功
- 目标 api field 应该存在
- 实际不存在
- 页面显示 fallback/空值

则命中。

```python
def hit_api_field_missing(ctx: DiagnosisContext) -> RuleFinding | None:
    if not ctx.response_success:
        return None

    if ctx.api_field_path is None:
        return None

    if ctx.api_field_exists:
        return None

    if ctx.displayed_value not in ctx.fallback_tokens:
        return None

    return RuleFinding(
        rule_code="R001",
        title="接口缺少页面所需字段",
        category="data_source",
        severity="high",
        confidence=0.92,
        layer="api",
        summary=f"接口成功返回，但缺少字段 {ctx.api_field_path}，页面最终显示 {ctx.displayed_value}",
        evidence_refs=["response_snapshot", "metadata_binding", "dom_snapshot"],
        suggestions=[
            "检查接口契约是否变更",
            "检查前后端字段映射是否一致",
            "增加 response schema 校验"
        ]
    )
```

```python
def hit_falsy_value_swallowed(ctx: DiagnosisContext) -> RuleFinding | None:
```

# 5. 规则冲突处理伪代码

因为有些规则会同时命中，所以要做去重和排序。

---

## 5.1 具体规则优先

例如：
- `R201 FALSY_VALUE_SWALLOWED`
- `R202 FORMATTER_OUTPUT_FALLBACK`

两者都命中时，优先保留更具体的 `R201`。

```python
SPECIFICITY_ORDER = {
    "R201": 100,
    "R202": 80,
    "R001": 95,
    "R003": 70,
    "R101": 90,
    "R403": 75,
}

def deduplicate_findings(findings: list[RuleFinding]) -> list[RuleFinding]:
    findings = sorted(
        findings,
        key=lambda x: (SPECIFICITY_ORDER.get(x.rule_code, 0), x.confidence),
        reverse=True
    )

    kept = []
    seen_categories = set()

    for f in findings:
        # 示例：相同问题簇内保留更具体的
        cluster = rule_cluster(f.rule_code)
        if cluster in seen_categories:
            continue
        seen_categories.add(cluster)
        kept.append(f)

    return kept
```

---

## 5.2 根因优先于表现

```python
ROOT_CAUSE_PRIORITY = {
    "api": 100,
    "state": 90,
    "selector": 85,
    "render": 80,
    "dom": 70,
    "ui": 60,
}

def rerank_findings(findings: list[RuleFinding], ctx: DiagnosisContext) -> list[RuleFinding]:
    def score(f: RuleFinding):
        return (
            ROOT_CAUSE_PRIORITY.get(f.layer, 0) * 0.4
            + f.confidence * 100 * 0.6
        )

    return sorted(findings, key=score, reverse=True)
```

## 6.1 页面值异常诊断决策树

```text
[开始: 页面显示异常值]
        │
        ▼
[是否拿到 response?]
   ├─ 否 → 不进入 API 规则，转查 click/request 链路
   └─ 是
        │
        ▼
[目标字段是否存在?]
   ├─ 否 → 命中 R001 API_FIELD_MISSING
   └─ 是
        │
        ▼
[字段值是否为 null/undefined?]
   ├─ 是 → 命中 R002 API_VALUE_NULL
   └─ 否
        │
        ▼
[store 是否更新?]
   ├─ 否 → 命中 R101 STORE_NOT_UPDATED
   └─ 是
        │
        ▼
[selector 是否输出空?]
   ├─ 是 → 命中 R103 SELECTOR_RETURNS_EMPTY
   └─ 否
        │
        ▼
[render 是否触发?]
   ├─ 否 → 命中 R102 / R203
   └─ 是
        │
        ▼
[render 输入是否为 0/false/"" 且输出 fallback?]
   ├─ 是 → 命中 R201 FALSY_VALUE_SWALLOWED
   └─ 否
        │
        ▼
[formatter 是否输出 fallback?]
   ├─ 是 → 命中 R202 FORMATTER_OUTPUT_FALLBACK
   └─ 否
        │
        ▼
[DOM 是否更新?]
   ├─ 否 → 命中 R301 DOM_NOT_UPDATED
   └─ 是
        │
        ▼
[DOM 是否可见?]
   ├─ 否 → 命中 R302 DOM_HIDDEN_OR_INVISIBLE
   └─ 是
        │
        ▼
[最终 DOM 是否 fallback 覆盖?]
   ├─ 是 → 命中 R303 DOM_OVERWRITTEN_BY_FALLBACK
   └─ 否 → 无明确规则命中，返回 INSUFFICIENT_EVIDENCE
```

## 6.2 点击没反应诊断决策树

```text
[开始: 用户点击后没反应]
        │
        ▼
[是否检测到 click_event?]
   ├─ 否 → 证据不足
   └─ 是
        │
        ▼
[是否有 handler_start?]
   ├─ 否 → 命中 R401 CLICK_NO_HANDLER
   └─ 是
        │
        ▼
[是否发出 request?]
   ├─ 否
   │    └─ [是否有 state_change?]
   │          ├─ 否 → 命中 R402 HANDLER_NO_REQUEST_NO_STATE_CHANGE
   │          └─ 是 → 继续检查 render
   └─ 是
        │
        ▼
[request 是否成功返回?]
   ├─ 否 → 非本批规则，交给网络异常类规则
   └─ 是
        │
        ▼
[是否有 state/render/dom 后续变化?]
   ├─ 否 → 命中 R403 REQUEST_SUCCESS_BUT_PAGE_NO_CHANGE
   └─ 是
        │
        ▼
[render 是否未发生?]
   ├─ 是 → 命中 R203 RENDER_NOT_TRIGGERED
   └─ 否
        │
        ▼
[DOM 是否未更新?]
   ├─ 是 → 命中 R301 DOM_NOT_UPDATED
   └─ 否 → 页面主链路成立，需进一步检查业务逻辑
```

# 8. MVP 先做哪几条的判定逻辑

建议第一批只先实现这 6 条的真判定：

- `R001 API_FIELD_MISSING`
- `R101 STORE_NOT_UPDATED`
- `R201 FALSY_VALUE_SWALLOWED`
- `R202 FORMATTER_OUTPUT_FALLBACK`
- `R203 RENDER_NOT_TRIGGERED`
- `R403 REQUEST_SUCCESS_BUT_PAGE_NO_CHANGE`

因为这 6 条：
- 最好解释
- 最容易从现有证据推出
- 最能体现产品价值

# 9. 一句话总结

这版伪代码 / 决策树的核心思路是：

> 把问题沿着  
> **API → Store → Selector → Render → DOM**  
> 和  
> **Click → Handler → Request → Response → State → Render → DOM**  
> 一层层往下判。

这样规则引擎会非常像一个“结构化排障专家”，而不是一堆分散 if-else。
