下面直接给你一版 **前端目录结构 + 模块职责草案**。  
我按你现在确定的路线来设计：

> **浏览器插件主采集 + 页面轻桥接补语义**

目标是：

- 低侵入
- MVP 可跑
- 后续可平滑扩展
- 不先做重型 SDK

---

# 1. 前端整体工程建议

我建议前端先拆成 **两个包**，而不是全塞一个目录：

```text
frontend/
  ├─ extension-app/        # 浏览器插件
  └─ page-bridge/          # 页面轻桥接 helper（超轻量）
```

这样好处很明显：

- 插件是主产品，独立演进
- 页面桥接是超轻依赖，方便业务页面按需引入
- 后面如果桥接升级成轻量 SDK，也有自然演进空间

# 4. 各模块职责说明

---

## 4.1 `background/`
这是插件的后台协调层，负责全局状态和通信，不直接做页面采集。

### `background/index.ts`
**职责：**
- 插件后台入口
- 初始化消息监听
- 初始化 tab/session 状态恢复
- 注册 popup / sidepanel / content script 通信

---

### `background/session-manager.ts`
**职责：**
- 管理诊断 session 生命周期
- 维护 session 状态：
  - idle
  - recording
  - stopped
  - uploading
  - uploaded
  - failed
- 按 tab 维度管理当前 session

**为什么重要：**  
录制状态最好不要由 popup 单独维护，否则 popup 一关状态就丢。

---

### `background/uploader.ts`
**职责：**
- 接收已停止的 session
- 调用后端上传 API
- 处理上传成功/失败
- 返回 diagnosisId
- 支持失败重试

---

### `background/tab-registry.ts`
**职责：**
- 维护 tabId 与页面会话的映射
- 识别当前活跃 tab
- 清理关闭 tab 的残留状态

---

### `background/message-router.ts`
**职责：**
- 统一处理各端消息：
  - popup -> background
  - content -> background
  - sidepanel -> background

这样可以避免消息协议散落在各处。

---

# 4.2 `content/`
这是插件真正的页面侧采集层，MVP 最核心。

---

### `content/index.ts`
**职责：**
- content script 入口
- 初始化 recorder
- 启动各类 observer
- 建立与 background 的连接

---

### `content/recorder.ts`
**职责：**
- 统一接收各 observer 产出的事件
- 把事件追加到当前 session buffer
- 控制录制开始/暂停/结束
- 对事件做最初级归并与时间线管理

它是 content 层的核心协调器。

---

### `content/dom-observer.ts`
**职责：**
- 监听用户关键交互：
  - click
  - submit
  - change
- 提取最小必要目标信息：
  - targetId
  - targetName
  - dom path 摘要
  - 文本摘要（裁剪后）

---

### `content/network-observer.ts`
**职责：**
- hook fetch / XHR
- 记录：
  - request start
  - response end
  - error
  - timeout（可选）
- 提取：
  - method
  - url pattern
  - status
  - duration
  - requestId

这是诊断链里非常重要的一层。

---

### `content/error-observer.ts`
**职责：**
- 监听：
  - `window.onerror`
  - `unhandledrejection`
  - resource error
- 记录错误摘要
- 短时间同类错误去重

---

### `content/route-observer.ts`
**职责：**
- 感知 SPA 路由变化
- 记录 route transition
- 辅助判断一次诊断发生在哪个页面阶段

---

### `content/bridge-listener.ts`
**职责：**
- 监听页面桥接协议：
  - `window.__DIAGNOSIS_CONTEXT__`
  - `diagnosis:interaction`
  - `diagnosis:state`
- 将桥接语义转换为标准 session event

这是“插件 + 轻桥接”路线的关键连接点。

---

### `content/ui-symptom-detector.ts`
**职责：**
- 识别页面可见症状线索
- MVP 只做轻量规则：
  - error toast
  - empty state
  - long loading
  - disabled primary button

**建议：**
不要一开始做太智能，先做有限规则识别。

---

### `content/event-buffer.ts`
**职责：**
- 内存事件缓存
- 限制队列大小
- 按优先级保留高价值事件
- 为 recorder 提供 append / snapshot / clear 能力

---

### `content/page-context-reader.ts`
**职责：**
- 读取当前页面基础上下文：
  - url
  - title
  - route
  - 页面 bridge context
- 在录制开始时生成 session 初始 pageContext

---

# 4.3 `popup/`
MVP 用户主入口。

---

### `popup/PopupApp.tsx`
**职责：**
- 渲染 popup 主界面
- 显示当前 tab/session 状态
- 提供：
  - 开始录制
  - 停止录制
  - 提交
  - 查看结果

---

### `popup/components/RecordButton.tsx`
**职责：**
- 展示录制主按钮
- 根据状态切换文案：
  - 开始诊断
  - 结束诊断
  - 正在上传

---

### `popup/components/SessionStatusCard.tsx`
**职责：**
- 展示当前状态
- 当前页面是否已连接 content script
- 当前 session 基础信息

---

### `popup/components/EventCounter.tsx`
**职责：**
- 展示已采集事件数量
- 简单分类计数：
  - interaction
  - network
  - error
  - bridge

---

### `popup/components/SubmitResultCard.tsx`
**职责：**
- 展示上传结果
- 显示 diagnosisId
- 提供“打开工作台”入口

---

### `popup/store/popup-store.ts`
**职责：**
- popup 层本地 UI 状态
- 从 background 拉 session 状态
- 处理 loading / error / result

---

# 4.4 `sidepanel/`
这个是推荐项，不是 MVP 必做，但很适合你产品形态。

---

### `sidepanel/SidePanelApp.tsx`
**职责：**
- 作为更完整的诊断会话视图
- 展示当前 timeline、页面上下文、用户备注

---

### `sidepanel/components/SessionTimeline.tsx`
**职责：**
- 时间线方式展示 session 中的关键事件
- 让用户更直观看到：
  - 先点了什么
  - 后发了什么请求
  - 最后出了什么错

这很贴你因果链产品方向。

---

### `sidepanel/components/EventPreviewList.tsx`
**职责：**
- 按类别显示最近采集的事件摘要

---

### `sidepanel/components/UserHintForm.tsx`
**职责：**
- 用户手动补充：
  - 问题描述
  - 预期结果
  - 实际结果

这对后端 explanation / AI summary 很有价值。

---

### `sidepanel/components/ContextSummary.tsx`
**职责：**
- 显示 page context / module / route / releaseVersion 等信息

---

# 4.6 `shared/`
这是整个前端工程的公共基础层，建议尽早规范。

---

## `shared/types/`
统一类型定义，避免 popup/content/background 各写一份。

### `diagnosis-session.ts`
定义：
- `DiagnosisSession`
- `PageContext`
- `UserHint`
- `SessionStatus`

### `probe-event.ts`
定义：
- `ProbeEvent`
- `UiEvent`
- `NetworkEvent`
- `ErrorEvent`
- `UiStateEvent`

### `bridge-event.ts`
定义：
- `BridgeContextEvent`
- `BridgeInteractionEvent`
- `BridgeStateEvent`

### `runtime-message.ts`
定义插件内部消息协议：
- `START_RECORDING`
- `STOP_RECORDING`
- `GET_SESSION_STATUS`
- `UPLOAD_SESSION`
- `SESSION_UPDATED`

### `api.ts`
定义与后端交互的 request/response 类型

---

## `shared/constants/`
放事件名、存储 key、bridge channel 等。

---

## `shared/utils/`
建议至少有：

### `id.ts`
- 生成 sessionId / eventId / requestId

### `sanitize.ts`
- 文本截断
- URL 脱敏
- 错误信息脱敏

### `url.ts`
- 规范化 URL
- 提取 pattern
- 过滤 query

---

## `shared/storage/`
统一 chrome storage / local state 访问方式。

---

## `shared/api/diagnosis-api.ts`
**职责：**
- 封装：
  - 上传 diagnosis session
  - 查询 diagnosis 结果
- 隔离 fetch 细节
- 给 background/uploader 直接使用

---

## `shared/state/session-machine.ts`
建议抽象一个轻量状态机，避免状态流乱掉。

例如：

```ts
type SessionStatus =
  | 'idle'
  | 'recording'
  | 'stopped'
  | 'uploading'
  | 'uploaded'
  | 'failed';
```

## 5.1 各文件职责

### `index.ts`
统一导出 bridge API

---

### `context.ts`
提供：

- `setDiagnosisContext`
- `getDiagnosisContext`

示例：

```ts
export function setDiagnosisContext(context: DiagnosisBridgeContext) {
  (window as any).__DIAGNOSIS_CONTEXT__ = context;
}
```

---

### `events.ts`
提供：

- `emitDiagnosisInteraction`
- `emitDiagnosisState`

示例：

```ts
export function emitDiagnosisInteraction(detail: DiagnosisInteractionDetail) {
  window.dispatchEvent(
    new CustomEvent('diagnosis:interaction', { detail }),
  );
}
```

---

### `types.ts`
定义桥接数据结构：

- `DiagnosisBridgeContext`
- `DiagnosisInteractionDetail`
- `DiagnosisStateDetail`

---

### `utils.ts`
放一些安全辅助：
- 过滤 undefined
- 序列化 detail
- 长度截断

# 6. 业务页面接入姿势建议

你后面给业务团队时，可以要求只接这两类东西：

---

## 6.1 页面初始化时设置上下文

```ts
setDiagnosisContext({
  appId: 'erp-console',
  module: 'order',
  page: '/orders/create',
  tenantId: tenantId,
  releaseVersion: version,
});
```

---

## 6.2 关键动作时发业务语义事件

```ts
emitDiagnosisInteraction({
  businessAction: 'submitOrder',
  targetId: 'submit-order-btn',
  targetName: '提交订单',
});
```

```ts
emitDiagnosisState({
  type: 'loading',
  businessAction: 'submitOrder',
  componentName: 'OrderSubmitPanel',
});
```

这已经足够轻了。

# 9. 最终建议

如果你现在要一个 **可落地、低侵入、面向 MVP 的前端结构**，我建议就定为：

## 插件主工程
- `background`：全局状态与上传
- `content`：页面采集核心
- `popup`：用户操作入口
- `shared`：类型、协议、工具

## 页面桥接轻包
- `context`
- `events`
- `types`

这样结构清楚，而且不会一开始就走重。
