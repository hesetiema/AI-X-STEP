# 一、统一模板建议

适用于已经进入 **matcher + factory** 形态的 rule。

---

## 通用模板

```ts
import { XxxRule } from '../../../src/diagnosis/domain/rule-engine/rules/xxx.rule';
import { buildDiagnosisContext } from '../../helpers/diagnosis-context.fixture';

describe('XxxRule', () => {
  let rule: XxxRule;

  beforeEach(() => {
    rule = new XxxRule();
  });

  it('should hit when <positive condition>', () => {
    const context = buildDiagnosisContext({
      evidence: [
        // minimal evidence for hit
      ],
    });

    const result = rule.evaluate(context);

    expect(result).toHaveLength(1);
    expect(result[0].ruleCode).toBe('Rxxx');
    expect(result[0].layer).toBe('...');
    expect(result[0].cluster).toBe('...');
    expect(result[0].isSymptomOnly).toBe(false); // or true
    expect(result[0].evidenceIds).toEqual(['ev-1']);
  });

  it('should not hit when <negative condition>', () => {
    const context = buildDiagnosisContext({
      evidence: [
        // minimal evidence for miss
      ],
    });

    const result = rule.evaluate(context);

    expect(result).toEqual([]);
  });
});
```

# 二、推荐断言边界

## 应该测
- 返回长度
- rule code
- layer / cluster
- symptom 属性
- evidenceIds 是否来自 matched evidence
- 某些规则独有 detail（仅在 detail 承载业务意义时）

## 不建议测太死
- `title`
- `summary`
- `confidence`
- `score`
- `rule.name`
- `rule.severity`

因为这些在 MVP 调整时更容易变化。  
它们更适合：
- 少量 smoke assertion
- 或 factory 单测统一覆盖

# 三、推荐辅助断言小工具（可选）

如果你想进一步统一 rule unit test，可以加一个 helper，例如：

## `test/helpers/rule-unit-assertions.ts`

```ts
import { RuleFinding } from '../../src/diagnosis/domain/types/diagnosis.types';

export function expectSingleRuleHit(
  result: RuleFinding[],
  expected: {
    ruleCode: string;
    layer: RuleFinding['layer'];
    cluster: RuleFinding['cluster'];
    isSymptomOnly: boolean;
    evidenceIds?: string[];
  },
): void {
  expect(result).toHaveLength(1);

  const finding = result[0];

  expect(finding.ruleCode).toBe(expected.ruleCode);
  expect(finding.layer).toBe(expected.layer);
  expect(finding.cluster).toBe(expected.cluster);
  expect(finding.isSymptomOnly).toBe(expected.isSymptomOnly);

  if (expected.evidenceIds !== undefined) {
    expect(finding.evidenceIds).toEqual(expected.evidenceIds);
  }
}

export function expectRuleMiss(result: RuleFinding[]): void {
  expect(result).toEqual([]);
}
```

# 五、这套“factory/matcher 时代”的 Rule 单测规范

我建议你后续统一按下面分型来写。

---

## A. 普通命中型规则
例如：
- `R201`
- `R202`
- `R203`
- `R302`
- `R401`
- `R402`
- `R503`
- `R504`

### 推荐 2 case
1. hit
2. miss

---

## B. 多入口命中型规则
例如：
- `R501`（evidence / symptom flag）
- 将来如果某条 rule 支持多个 matcher path，也归这类

### 推荐 3 case
1. hit by evidence
2. hit by alternate signal
3. miss when all paths absent

---

## C. 聚合型/优先级型 rule
如果将来某条 rule：
- 需要组合多种 evidence
- 或存在更复杂判断

那 rule unit test 至少补：
1. primary hit
2. secondary hit
3. miss
4. optional detail assertion
