# 2. 核心闭环架构图

你这个产品最关键的不是“采集”，而是下面这个闭环：

```text
页面一个点 / 一个异常
        ↓
前端探针定位组件、值、状态、请求
        ↓
Trace Builder 拼出真实交互链
        ↓
静态图谱 + 运行时图谱合并
        ↓
血缘引擎做正向/反向追踪
        ↓
规则引擎先给确定性结论
        ↓
AI 引擎做归因解释、修复建议、测试建议
        ↓
工作台给出可视化因果图 + 诊断报告
```

# 3. 分层详细说明

---

## 3.1 业务前端应用层

这是被观测对象。

### 典型组成
- React/Vue 页面
- 路由
- 状态管理
- API 调用层
- 数据转换层
- 组件渲染层
- 埋点/日志逻辑

### 关键要求
尽量 **低侵入接入**，否则很难推广。

### 接入方式建议
1. **浏览器插件模式**
   - 最适合开发/测试环境
   - 无需业务代码强接入
   - 适合先验证 MVP

2. **SDK 模式**
   - 用于线上或灰度环境
   - 可控采集
   - 便于长期产品化

3. **构建期自动插桩**
   - Babel / SWC / Vite / Webpack 插件
   - 给特定函数、组件、selector 自动加标识

---

## 3.2 前端探针采集层

这是整个平台最底座的能力层。

### 目标
收集“页面上到底发生了什么”。

---

### 子模块说明

#### 1）Event Probe
采集：
- click
- input
- change
- submit
- route change
- custom business event

记录：
- target DOM
- target component
- timestamp
- interaction_id
- event payload

---

#### 2）Network Probe
采集：
- 请求 URL / method / headers
- request payload
- response payload
- status code
- latency
- error
- trace id

额外能力：
- 结构摘要
- 字段存在性
- 字段类型分布
- 契约漂移候选

---

#### 3）State Probe
采集：
- action dispatch
- store patch
- selector 输出变化
- query cache 更新
- context value 变化

支持：
- Redux
- Zustand
- MobX
- Vuex
- Pinia
- React Query / SWR 等

---

#### 4）Render Probe
采集：
- 哪个组件重新渲染了
- render 原因
- render 前后 props/state 摘要
- render duration
- commit 时间

---

#### 5）DOM Probe
采集：
- 节点是否存在
- 文本内容变化
- 可见性变化
- 属性变化
- fallback 渲染

---

#### 6）Transform Probe
用于采集最关键的中间层：
- formatter
- mapper
- normalize
- adapter
- selector
- derived state

这个模块非常重要，因为很多问题就发生在这里。

---

#### 7）Inspect Agent
支持在页面上点一个值，直接定位：
- DOM 节点
- 组件
- 绑定表达式
- props/store/query
- 对应的 trace

这部分是你产品最有差异化的入口。

---

## 3.3 Trace Ingestion & Session 层

这一层把前端的“碎片事件”变成“可理解的一次交互”。

### 核心模块

#### 1）Event Gateway
接前端上报数据：
- 实时 websocket
- batched HTTP
- 本地缓冲后上传

---

#### 2）Session Manager
维护上下文：
- user session
- page session
- route session
- interaction context

---

#### 3）Trace Builder
把这些碎片拼起来：
- 点击事件
- 后续请求
- 状态更新
- 渲染变化
- DOM 更新

产出一个 `interaction trace`。

---

#### 4）Correlation Engine
做事件关联：
- 这个请求是不是由那个按钮触发的
- 这个组件渲染是不是由那个 store 更新导致的
- 这个 DOM 文本变化是不是来自某个 transform 结果

---

#### 5）Sampling Controller
控制性能成本：
- 默认低采样
- 异常时高精度录制
- 开发环境全量
- 线上灰度采样

---

## 3.4 静态分析与代码图谱层

运行时看“发生了什么”，静态分析看“理论上可能发生什么”。

### 核心目的
建立系统结构语义，补足 runtime 看不到的关系。

### 分析对象
- 组件树
- props 传递
- hooks 调用
- selector
- action/store
- API 调用函数
- formatter
- 路由绑定
- 条件渲染逻辑
- git 变更

### 产出图谱
- Component Graph
- Route Graph
- Store Dependency Graph
- API Mapping Graph
- Field Transform Candidates

---

## 3.5 运行时血缘与依赖图层

这是你的核心壁垒。

### 目标
回答两个问题：

1. **点击这里后发生了什么**
2. **这个值是从哪里来的**

### 关键能力

#### Runtime Dependency Graph
建立这次真实运行中实际发生的依赖图。

#### Field Lineage Resolver
字段级血缘解析：
- API response.data.a
- normalizeA()
- selectorB()
- formatterC()
- componentD
- DOM text

#### Reverse Trace Resolver
从页面展示值反向找源头。

#### API→State→View Path Builder
构建非常直观的路径：
`API字段 -> Store字段 -> Selector -> Formatter -> Component -> DOM`

---

## 3.6 Unified Knowledge Graph 层

建议你最终统一落在一个“知识图谱/依赖图谱”表示层。

### 为什么需要
因为你后面的：
- 根因分析
- 影响分析
- AI 问答
- 回归测试生成

都需要统一的结构化表示。

### 节点举例
- Page
- Route
- Component
- DOM Node
- Event
- API
- Response Field
- Store
- Selector
- Formatter
- Render
- Error
- Commit

### 边举例
- triggers
- updates
- reads_from
- transforms_to
- renders
- hidden_by
- blocked_by
- sourced_from
- affected_by

---

## 3.7 规则诊断与检测引擎

这层建议一定要有，不能全靠 AI。

因为很多前端异常其实是高确定性的规则问题。

### 建议首批规则
- `if (!value)` 把 0 过滤掉
- null/undefined 经 formatter 后变空串
- 接口字段缺失
- store 更新但 selector 输出没变
- selector 变了但组件未刷新
- 请求成功但无消费者
- fallback 覆盖真实值
- 缓存未失效导致旧值展示
- 列表 key 异常导致渲染错位

### 作用
- 先给稳定、可验证的归因
- 再把规则结论喂给 AI 做解释和排序

---

## 3.8 AI 分析引擎

AI 不是底座，而是增强层。

### 关键 Agent 能力

#### 1）Trace Summarizer
把 200 个事件总结成一句话。

#### 2）Root Cause Analyzer
结合规则结果、图谱、trace 给根因排序。

#### 3）Field Lineage Explainer
把字段路径用人话解释清楚。

#### 4）Fix Suggestor
建议代码层面的修复。

#### 5）Test Case Generator
生成：
- UI 回归测试
- selector 单测
- formatter 单测

#### 6）Instrumentation Planner
告诉开发哪里该加埋点、断点、监控。

#### 7）Change Impact Analyst
结合 git diff 看最近改动是否影响这个展示值。

---

## 3.9 产品工作台 / 可视化层

这是用户真正感知的平台。

### 核心页面建议

#### 1）页面 Inspect 面板
从页面中选一个值，查看：
- 当前值
- 组件
- 数据来源
- 最近链路
- 异常提示

#### 2）点击因果链时间线
展示点击后的全过程。

#### 3）多米诺回放动画
适合演示、培训、排查。

#### 4）字段级血缘图
最核心的分析视图之一。

#### 5）异常诊断报告页
给出：
- 根因排序
- 证据链
- 修复建议
- 测试建议

#### 6）AI 问答 Copilot
自然语言调试入口。

   ↓

# 7. 你可以拿去和团队沟通的“核心架构原则”

我帮你提炼成几条，讨论时很好用：

### 原则 1：从页面上的一个点开始
架构设计不是从日志系统出发，而是从页面元素和展示值出发。

### 原则 2：静态 + 运行时双图结合
只做静态图不准，只做运行时图不全，必须融合。

### 原则 3：规则优先，AI 增强
先用规则引擎给稳定判断，再由 AI 总结和建议。

### 原则 4：字段级血缘优先于请求级链路
真正解决前端问题的是字段怎么变成展示值，而不是只看请求是否成功。

### 原则 5：低侵入、可灰度、可采样
没有低侵入接入，这个平台就很难推广。
