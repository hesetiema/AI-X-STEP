下面直接给你 **RuleFinding factory**，以及基于它的 **4 条 rule 再压缩版**。

这一步的目标很明确：

- 把 rule 文件里重复的大对象拼装收敛掉
- 让 rule 文件只保留：
  1. 收集 matched evidence
  2. 没命中直接 `[]`
  3. 调 `buildRuleFindingFromMatch(...)`

这样后面 rule 数量继续增加时，维护成本会明显下降。

# 二、`buildRuleFindingFromMatch` 首版

## `src/diagnosis/domain/rule-engine/factories/build-rule-finding.ts`

```ts
import {
  DiagnosisContext,
  RuleFinding,
} from '../../types/diagnosis.types';

type EvidenceItem = DiagnosisContext['evidence'][number];

type BuildRuleFindingInput = {
  ruleCode: string;
  ruleName: string;
  title: string;
  summary: string;
  confidence: number;
  score: number;
  layer: RuleFinding['layer'];
  cluster: RuleFinding['cluster'];
  severity: NonNullable<RuleFinding['rule']>['severity'];
  isSymptomOnly: boolean;
  matchedEvidence: EvidenceItem[];
  detail?: Record<string, unknown>;
};

export function buildRuleFindingFromMatch(
  input: BuildRuleFindingInput,
): RuleFinding {
  const {
    ruleCode,
    ruleName,
    title,
    summary,
    confidence,
    score,
    layer,
    cluster,
    severity,
    isSymptomOnly,
    matchedEvidence,
    detail,
  } = input;

  return {
    ruleCode,
    title,
    summary,
    confidence,
    score,
    layer,
    cluster,
    evidenceIds: matchedEvidence.map((x) => x.id),
    isSymptomOnly,
    detail: {
      matchedCount: matchedEvidence.length,
      ...(detail ?? {}),
    },
    rule: {
      code: ruleCode,
      name: ruleName,
      cluster,
      layer,
      severity,
      isSymptomOnly,
    },
  };
}
```

# 三、为什么先做这个版本

这个 factory 先只做一件事：

- **把标准 RuleFinding 对象组装统一化**

暂时不做太多抽象，比如：
- 自动算 score
- 自动推导 severity
- 自动推导 title/summary
- 自动 merge context meta

原因是你现在还在 MVP 规则 hardening 阶段，  
太早把“策略”和“对象构建”绑死，会降低后续调整空间。

这个版本足够轻，也足够实用。

## 1）`src/diagnosis/domain/rule-engine/rules/r201-api-5xx.rule.ts`

import { Injectable } from '@nestjs/common';
import { DiagnosisRule } from '../interfaces/diagnosis-rule.interface';
import {
  DiagnosisContext,
  RuleFinding,
} from '../../types/diagnosis.types';
import {
  collectNetworkEvidence,
  isHttp5xxEvidence,
} from '../matchers/network.matcher';
import { buildRuleFindingFromMatch } from '../factories/build-rule-finding';

@Injectable()
export class R201Api5xxRule implements DiagnosisRule {
  readonly code = 'R201';

  evaluate(context: DiagnosisContext): RuleFinding[] {
    const matchedEvidence = collectNetworkEvidence(context, isHttp5xxEvidence);

    if (matchedEvidence.length === 0) {
      return [];
    }

    return [
      buildRuleFindingFromMatch({
        ruleCode: 'R201',
        ruleName: 'API 5xx indicates downstream instability',
        title: 'API 5xx root cause',
        summary:
          'Observed API 5xx response indicating backend/API-side instability.',
        confidence: 0.91,
        score: 91,
        layer: 'api',
        cluster: 'api_failure',
        severity: 'high',
        isSymptomOnly: false,
        matchedEvidence,
      }),
    ];
  }
}

# 五、压缩前后对比

## 压缩前 rule 的主要结构
每个 rule 都要手写：

- `ruleCode`
- `title`
- `summary`
- `confidence`
- `score`
- `layer`
- `cluster`
- `evidenceIds`
- `isSymptomOnly`
- `detail.matchedCount`
- `rule.code`
- `rule.name`
- `rule.cluster`
- `rule.layer`
- `rule.severity`
- `rule.isSymptomOnly`

很重复。

---

## 压缩后
每个 rule 现在只剩三段：

### 1. collect
```ts
const matchedEvidence = ...
```

### 2. guard
```ts
if (matchedEvidence.length === 0) {
  return [];
}
```

### 3. build
```ts
return [buildRuleFindingFromMatch({...})];
```

这就非常适合大规模规则扩展。

# 六、如果你愿意，再进一步可以做什么

## 方向 A：让 factory 自动补 matchedCount 以外的 detail
比如支持：

```ts
detail: {
  targetId: context.targetId,
}
```

现在已经支持，因为 `detail` 会 merge 到结果里：

```ts
detail: {
  matchedCount: matchedEvidence.length,
  ...(detail ?? {}),
}
```

例如你可以这样写：

```ts
buildRuleFindingFromMatch({
  ...
  matchedEvidence,
  detail: {
    targetId: context.targetId,
  },
})
```

---

## 方向 B：加一个 batch helper，进一步减少 rule 文件样板
比如做一个：

```ts
buildSingleFindingOrEmpty({
  matchedEvidence,
  config: {...}
})
```

这样 rule 里连 `if (matchedEvidence.length === 0)` 都不用写。

但我建议先别急。  
你现在这个版本已经在“减少重复”和“保持清晰”之间比较平衡。
