下面直接给你 **DiagnosisModule provider wiring 测试 / module compile spec**。  
目标是锁住这几个最容易在后续重构时悄悄坏掉的点：

- `DiagnosisModule` 能正常 compile
- `DiagnosisController` 能注入到真实 service
- `DIAGNOSIS_REPOSITORY` token 注册正确
- `'DiagnosisContextBuilder'` alias 注册正确
- `'DIAGNOSIS_RULES'` 聚合注入正确

这类测试非常值，因为它能防止：

- provider token 改名没同步
- useExisting / useFactory wiring 断掉

# 8. 建议同步修改 `DiagnosisModule`

下面给你一版 **字符串 token 全替换后的 module**，方便和上面的 spec 对齐。

## `src/modules/diagnosis/diagnosis.module.ts`

```ts
import { Module } from '@nestjs/common';

import { DiagnosisController } from './controller/diagnosis.controller';

import { DiagnosisCommandService } from './application/services/diagnosis-command.service';
import { DiagnosisQueryService } from './application/services/diagnosis-query.service';
import { DiagnosisContextBuilder } from './application/builders/diagnosis-context.builder';

import { DominoChainBuilder } from './domain/builders/domino-chain.builder';
import { ExplanationBuilder } from './domain/builders/explanation.builder';
import { DiagnosisConclusionService } from './domain/services/diagnosis-conclusion.service';
import { RankingService } from './domain/services/ranking.service';
import { RuleEngineService } from './domain/services/rule-engine.service';

import { DiagnosisRuleRegistry } from './domain/rules/registry/diagnosis-rule.registry';
import { R101ApiFieldMissingRule } from './domain/rules/r101-api-field-missing.rule';
import { R302DomNotUpdatedRule } from './domain/rules/r302-dom-not-updated.rule';
import { R501FallbackDisplayedRule } from './domain/rules/r501-fallback-displayed.rule';

import {
  DIAGNOSIS_CONTEXT_BUILDER,
  DIAGNOSIS_REPOSITORY,
  DIAGNOSIS_RULES,
} from './infrastructure/constants/diagnosis-repository.token';
import { InMemoryDiagnosisRepository } from './infrastructure/repositories/in-memory-diagnosis.repository';

const RULE_PROVIDERS = [
  R101ApiFieldMissingRule,
  R302DomNotUpdatedRule,
  R501FallbackDisplayedRule,
];

@Module({
  controllers: [DiagnosisController],
  providers: [
    DiagnosisCommandService,
    DiagnosisQueryService,
    DiagnosisContextBuilder,

    RuleEngineService,
    RankingService,
    DiagnosisConclusionService,
    ExplanationBuilder,
    DominoChainBuilder,

    DiagnosisRuleRegistry,
    ...RULE_PROVIDERS,

    {
      provide: DIAGNOSIS_CONTEXT_BUILDER,
      useExisting: DiagnosisContextBuilder,
    },
    {
      provide: DIAGNOSIS_RULES,
      useFactory: (
        r101: R101ApiFieldMissingRule,
        r302: R302DomNotUpdatedRule,
        r501: R501FallbackDisplayedRule,
      ) => [r101, r302, r501],
      inject: [
        R101ApiFieldMissingRule,
        R302DomNotUpdatedRule,
        R501FallbackDisplayedRule,
      ],
    },
    {
      provide: DIAGNOSIS_REPOSITORY,
      useClass: InMemoryDiagnosisRepository,
    },
  ],
  exports: [
    DiagnosisCommandService,
    DiagnosisQueryService,
    RuleEngineService,
    DiagnosisConclusionService,
    ExplanationBuilder,
    DominoChainBuilder,
  ],
})
export class DiagnosisModule {}
```

# 9. 这类 spec 的价值

这份 module wiring spec 主要防以下回归：

---

## 9.1 token 改了但注入点没改
比如：

- module 里写 `DiagnosisContextBuilder`
- service 里写 `'DiagnosisContextBuilder'`
- 某天你改成常量但漏一处

这种 compile spec 会第一时间炸。

---

## 9.2 新增规则但忘记放进 DIAGNOSIS_RULES 聚合
例如加了 `R103StateNotUpdatedRule` provider，但没写进 factory return。  
那 rule class 虽然存在，`RuleEngineService` 实际上跑不到。  
这类 bug 非常隐蔽，compile spec 能挡住。

---

## 9.3 useExisting alias 断了
如果以后你把：

```ts
provide: DIAGNOSIS_CONTEXT_BUILDER,
useExisting: DiagnosisContextBuilder,
```

误改成别的 provider，这条测试会发现 alias 不再指向同一个实例。

---

# 10. 一个更稳的增强版建议

如果你不想在测试里使用：

```ts
Reflect.get(...)
```

可以给部分 service 增加“仅测试用途”的受保护 getter，  
但我一般不建议为了测试改生产代码，所以当前用 `Reflect.get` 是可以接受的。
