# 2. 关键阶段说明

---

## 阶段 A：交互起点识别

### 发生什么
用户点击页面元素。

### 系统要记录什么
- 点击的是哪个 DOM
- 对应哪个组件
- 当前路由
- 当前用户会话
- 当前页面上下文
- interaction_id

### 作用
给这一整条链路一个“起点锚点”。

如果没有这个 interaction_id，后面请求、状态变化、渲染很难串起来。

---

## 阶段 B：事件处理函数执行

### 可能发生的事情
一个点击后通常会触发多种行为：

- 调用本地 handler
- dispatch action
- setState
- 发请求
- 路由跳转
- 埋点上报
- 打开弹窗
- 触发二次交互

### 你要关注的点
不仅要记录“函数执行了”，还要记录：

- 输入参数
- 返回结果
- 是否抛错
- 是否提前 return
- 是否异步分叉

---

## 阶段 C：副作用扩散

这是“多米诺”最明显的部分。

### 典型扩散路径
#### 路径 1：请求链
click → handler → request → response

#### 路径 2：状态链
click → dispatch → reducer/store patch → selector change

#### 路径 3：渲染链
state change → component rerender → DOM update

#### 路径 4：导航链
click → navigate → route change → page mount → request → render

#### 路径 5：组合链
click → request → normalize → store update → render → child render → DOM update

平台需要把这些路径合并成一条完整因果链。

---

## 阶段 D：变换与派生值采集

这一步非常关键，因为很多问题不在请求本身，而在中间转换。

### 需要采集的函数类型
- normalize
- adapter
- selector
- formatter
- mapper
- derived state
- computed hooks

### 为什么关键
接口返回是对的，但 UI 显示错，经常是这里出问题。

### 示例
```text
response.data.amount_cent = 10000
→ normalizeOrder()
→ store.currentOrder.amount = 10000
→ centToYuan(10000) = 100
→ formatCurrency(100) = "¥100.00"
→ DOM text 更新
```

这条链非常适合做多米诺动画。

---

## 阶段 E：渲染与 DOM 落地

最终用户真正感知的是页面变化，不是 store 更新。

### Render Probe 关注
- 哪些组件刷新了
- 为什么刷新
- 哪些应该刷新但没刷新
- 哪些发生了无效重渲染

### DOM Probe 关注
- 哪些文本变了
- 哪些区域出现/消失
- 哪些 class/style 变化
- 哪些 fallback 被显示

---

## 阶段 F：Trace 拼装与因果推断

### Trace Builder 负责
把原始采集事件整理成时间序列。

### Causality Engine 负责
把时间上相近的事件，推断成真正的“因果关系”。

不是简单按时间排序，而是要识别：
- 请求是不是由这次点击触发
- 渲染是不是由这次 state patch 导致
- DOM 更新是不是来自这个组件 commit

### 最终输出
一张可解释的交互因果图。

# 3. 异常/中断分支图

这个图更适合“点击后没反应”的场景。

```text
用户点击按钮
    │
    ▼
是否捕获到 click event？
 ├─ 否 → 事件绑定缺失 / 遮罩覆盖 / disabled / DOM未命中
 └─ 是
      │
      ▼
是否进入 handler？
 ├─ 否 → 绑定函数丢失 / stopPropagation / 条件阻断
 └─ 是
      │
      ▼
handler 是否提前 return / 抛错？
 ├─ 是 → 参数校验失败 / 运行时错误 / 守卫条件不满足
 └─ 否
      │
      ▼
是否产生副作用？
 ├─ 否 → 点击无业务逻辑 / 埋点后结束 / 漏写逻辑
 └─ 是
      │
      ├─ 发请求？
      │    ├─ 否 → 可能只是本地状态更新
      │    └─ 是
      │         │
      │         ▼
      │       请求是否成功？
      │         ├─ 否 → 接口异常 / 网络异常
      │         └─ 是
      │              │
      │              ▼
      │       响应是否进入 transform / store？
      │         ├─ 否 → 结果未消费 / Promise链中断
      │         └─ 是
      │
      └─ 更新状态？
           ├─ 否 → 没有任何状态变化
           └─ 是
                │
                ▼
           组件是否重渲染？
                ├─ 否 → memo阻断 / selector无变化 / 订阅丢失
                └─ 是
                     │
                     ▼
                DOM是否变化？
                     ├─ 否 → 渲染结果相同 / 条件渲染未命中
                     └─ 是 → 链路成功完成
```

# 4. 多米诺动画映射方式

你前面提的“像多米诺骨牌一样展示数据流”，这张时序图正好可以映射成动画节点。

---

## 4.1 动画节点类型建议

每一步都作为一个卡片/骨牌节点：

- 用户点击
- handler 执行
- action dispatch
- request 发起
- response 返回
- transform 执行
- store 更新
- component render
- DOM update

---

## 4.2 动画播放逻辑

### 模式 1：严格按时间播放
适合回放。

```text
Click
 → Request
   → Response
     → Store Update
       → Component Render
         → DOM Update
```

### 模式 2：按因果分支播放
适合复杂交互。

```text
Click
 ├─ 埋点事件
 ├─ 请求A
 │   └─ 更新模块A
 └─ 请求B
     └─ 更新模块B
```

### 模式 3：中断点高亮
适合排查。

```text
Click
 → Request
   → Response
     → Transform [失败/吞值]
       ✖ DOM 未更新
```

---

## 4.3 每个骨牌节点建议展示的信息

### Event 节点
- target
- component
- timestamp

### Handler 节点
- handler name
- 执行耗时
- 是否异常

### Request 节点
- url
- method
- status
- latency

### Transform 节点
- 函数名
- 输入摘要
- 输出摘要

### State 节点
- store path
- 变更前/后

### Render 节点
- component name
- render reason
- 耗时

### DOM 节点
- 文本变化
- 节点出现/隐藏
- 样式变化

# 7. MVP 最小闭环

如果你现在只做 MVP，我建议正向链路先只实现这 6 个节点：

```text
Click
 → Handler
   → Request
     → Response
       → Store Update
         → Render/DOM Update
```

先别一上来就做：
- 全量函数调用图
- 复杂异步任务图
- 多线程/worker 传播
- 跨页面全链路

MVP 做到下面这几件事就已经很强了：

1. 用户点一个按钮，能看到后续请求和更新链
2. 能标出链路中断在哪一步
3. 能看到最终哪些页面区域变了
4. 能自动总结一句人话解释

# 8. 这张图的产品价值总结

这张“点击交互正向链路时序图”本质上是在把：

> “点击之后没反应 / 不知道发生了什么”

变成：

> “一条可回放、可解释、可诊断的交互因果链”

它和你前面的“页面值反向溯源时序图”正好组成一对：

- **正向图**：点击后会发生什么
- **反向图**：这个值是从哪里来的

这两个是你整个平台最核心的双支柱。
