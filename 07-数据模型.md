# 2. 核心实体说明

MVP 阶段我建议只保留 9 个核心对象。  
够用，而且不会太重。

---

## 2.1 Page Session
表示一次页面会话上下文。

### 用途
把一段时间内的页面活动串起来。

### 为什么需要
同一个用户可能：
- 打开多个页面
- 同一路由多次进入
- 一次 interaction 需要知道所属页面上下文

### 示例
- 用户进入订单详情页
- 停留 10 分钟
- 期间点了刷新按钮、编辑按钮、查看日志按钮

这些 interaction 都属于一个或多个 page session。

---

## 2.2 Interaction
表示一次“可追踪交互”。

### 类型建议
- `click`
- `inspect`
- `route_change`
- `auto_refresh`（可选）
- `input_submit`（可选）

### 用途
它是 MVP 的主锚点。

#### 正向链路
一次 click interaction 下，挂所有 trace node。

#### 反向溯源
一次 inspect interaction 下，挂 lineage analysis 和 diagnosis。

---

## 2.3 Trace Node
表示链路里的一个事件节点。

### MVP 建议支持的 node_type
- `event`
- `handler`
- `network_request`
- `network_response`
- `state_update`
- `transform`
- `render`
- `dom_update`
- `error`

### 作用
它是时间线和因果图的基础。

---

## 2.4 Snapshot
表示某个节点的详细快照内容。

### 为什么要拆出来
Trace Node 表本身应该轻量，不要塞大 JSON。

### 例如
- network response body
- store before/after
- render props/state
- DOM text before/after

这些可以放到对象存储，DB 里只存引用。

---

## 2.5 Evidence
表示“诊断时会引用的证据摘要”。

### 和 Snapshot 的区别
- Snapshot 是原始或半原始数据
- Evidence 是为了诊断/展示提炼后的证据

### 例子
- “接口返回 `amount_cent = 0`”
- “formatter 输出 `--`”
- “store 中值已更新，但组件未重渲染”

Evidence 是给规则引擎和报告页用的。

---

## 2.6 Causality Edge
表示节点间因果关系。

### 为什么单独建模
时间相邻不等于因果相关。

需要明确表达：
- event 导致 request
- request response 导致 state update
- state update 导致 render
- render 导致 dom update

### relation 建议
- `triggers`
- `causes`
- `updates`
- `renders`
- `fails_before`
- `depends_on`

---

## 2.7 Lineage Analysis
表示一次页面值溯源任务。

### 用途
用户点了页面上一个值/区域后，系统生成一个 lineage 分析对象。

### 它和 Interaction 的关系
- 通常由 `inspect` interaction 触发
- 也可以附着在某次 `click` interaction 上的目标结果区域

---

## 2.8 Lineage Step
表示反向血缘路径中的一步。

### 典型路径
```text
DOM.text
← render expression
← component prop
← selector/store
← api field
```

每一层就是一个 step。

### 为什么拆开
这样：
- 可以做图形展示
- 可以做规则判断
- 可以支持多候选路径

---

## 2.9 Rule Finding
表示规则命中结果。

### 例子
- `API_FIELD_MISSING`
- `FALSY_VALUE_SWALLOWED`
- `RENDER_NOT_TRIGGERED`
- `DOM_HIDDEN`
- `RESPONSE_NOT_CONSUMED`

### 用途
给 Diagnosis Report 提供确定性依据。

---

## 2.10 Diagnosis Report
表示最终给用户看的诊断结果。

### 内容包括
- 问题摘要
- 根因排序
- 证据链
- 修复建议
- 置信度

---

# 3. 实体关系图

下面是更清晰的 ER 风格关系图。

```text
PageSession 1 ───── N Interaction
Interaction 1 ───── N TraceNode
TraceNode   1 ───── N Snapshot
TraceNode   1 ───── N Evidence

TraceNode   1 ───── N CausalityEdge(from)
TraceNode   1 ───── N CausalityEdge(to)

Interaction 1 ───── 0..N LineageAnalysis
LineageAnalysis 1 ─ N LineageStep

Interaction 1 ───── 0..N RuleFinding
LineageAnalysis 1 ─ 0..N RuleFinding

Interaction 1 ───── 0..1 DiagnosisReport
LineageAnalysis 1 ─ 0..1 DiagnosisReport
```

# 4. 关键字段设计

下面我给你一版偏工程可落地的字段建议。

---

## 4.1 page_session

```text
page_session
- session_id              PK
- user_id                 nullable
- app_id
- env                     dev/test/prod
- page_url
- route
- page_title
- user_agent
- started_at
- ended_at
- created_at
```

### 说明
MVP 里不需要把用户体系做复杂，`user_id` 可以为空。

---

## 4.2 interaction

```text
interaction
- interaction_id          PK
- session_id              FK -> page_session
- interaction_type        click / inspect / route_change
- target_dom_selector
- target_dom_text
- target_component
- trigger_source          user / auto / system
- started_at
- ended_at
- status                  success / failed / partial
- summary
- created_at
```

### 说明
这是主查询入口之一。

---

## 4.3 trace_node

```text
trace_node
- node_id                 PK
- interaction_id          FK -> interaction
- parent_node_id          nullable self FK
- node_type               event / handler / network_request / network_response / state_update / transform / render / dom_update / error
- node_name               如 onSubmit / GET /api/order/detail / OrderAmount render
- layer                   ui / handler / network / state / transform / render / dom
- component_name          nullable
- store_key               nullable
- api_name                nullable
- timestamp
- duration_ms             nullable
- status                  success / failed / skipped
- payload_ref             nullable
- summary                 文本摘要
- created_at
```

### 说明
`summary` 很有用，方便 UI 快速展示。

---

## 4.4 snapshot

```text
snapshot
- snapshot_id             PK
- node_id                 FK -> trace_node
- snapshot_type           request / response / state_before / state_after / render_props / render_output / dom_before / dom_after / transform_io
- content_ref             指向对象存储
- content_hash
- content_size
- created_at
```

### 说明
对象存储里放大 JSON。

---

## 4.5 evidence

```text
evidence
- evidence_id             PK
- node_id                 nullable FK -> trace_node
- interaction_id          FK -> interaction
- evidence_type           api_field / store_value / formatter_output / dom_visibility / render_reason / error_message
- evidence_key            如 amount_cent / order.total / formatCurrency
- value_summary
- raw_ref                 nullable
- created_at
```

### 说明
Evidence 是“面向诊断消费”的抽象层，建议保留。

---

## 4.6 causality_edge

```text
causality_edge
- edge_id                 PK
- interaction_id          FK -> interaction
- from_node_id            FK -> trace_node
- to_node_id              FK -> trace_node
- relation                triggers / causes / updates / renders / derived_to
- confidence              decimal(5,4)
- created_at
```

### 说明
置信度可以在 MVP 先简单一点，默认固定值也行。

---

## 4.7 lineage_analysis

```text
lineage_analysis
- lineage_id              PK
- interaction_id          nullable FK -> interaction
- inspect_target_id       nullable
- target_dom_selector
- target_component
- displayed_value
- expected_value          nullable
- analysis_status         pending / resolved / failed
- created_at
- updated_at
```

### 说明
`expected_value` MVP 可选，不一定都有。

---

## 4.8 lineage_step

```text
lineage_step
- step_id                 PK
- lineage_id              FK -> lineage_analysis
- step_order              int
- direction               reverse / forward
- layer                   dom / render / component / state / transform / api
- source_type             dom_text / render_expr / prop / selector / store_key / api_field / formatter
- source_key              如 formatCurrency(amount) / order.current.amount_cent
- component_name          nullable
- value_summary           如 "--" / 0 / null / "¥0.00"
- evidence_ref            nullable
- created_at
```

### 说明
这张表基本就能把“反向血缘图”画出来。

---

## 4.9 rule_finding

```text
rule_finding
- finding_id              PK
- interaction_id          nullable FK -> interaction
- lineage_id              nullable FK -> lineage_analysis
- rule_code               API_FIELD_MISSING / FALSY_VALUE_SWALLOWED / RENDER_NOT_TRIGGERED / DOM_HIDDEN / RESPONSE_NOT_CONSUMED
- severity                low / medium / high
- confidence              decimal(5,4)
- title
- description
- evidence_refs           json
- created_at
```

### 说明
`evidence_refs` 可以先用 JSON，后续再做关系规范化。

---

## 4.10 diagnosis_report

```text
diagnosis_report
- report_id               PK
- interaction_id          nullable FK -> interaction
- lineage_id              nullable FK -> lineage_analysis
- report_type             inspect_diagnosis / click_trace_summary
- summary
- root_cause_top1
- root_cause_ranking      json
- confidence              decimal(5,4)
- suggestions             json
- ai_summary              text
- status                  ready / failed
- created_at
- updated_at
```

### 说明
MVP 阶段允许一些聚合结果直接 JSON 化，别过度规范化。

# 7. MVP 存储建议

MVP 阶段不要过度设计，建议：

---

## 7.1 关系型数据库
适合存：
- page_session
- interaction
- trace_node
- causality_edge
- lineage_analysis
- lineage_step
- rule_finding
- diagnosis_report

建议：
- PostgreSQL 或 MySQL 都可以
- 如果要查 JSON 更灵活，PostgreSQL 更舒服

---

## 7.2 对象存储
适合存：
- request/response body
- render props/state 大快照
- DOM before/after
- transform input/output 大对象

建议：
- S3 / MinIO / OSS

---

## 7.3 可选缓存
适合：
- 最近 interaction 查询
- 最近页面 session 上下文
- inspect 临时结果

建议：
- Redis

---

# 9. 一句话总结这个模型

这套 MVP 数据模型的核心思想是分四层：

- **Session / Interaction**：定义上下文
- **Trace Node / Snapshot**：记录运行时事实
- **Lineage / Causality / Evidence**：建立链路关系
- **Rule Finding / Diagnosis Report**：输出诊断结果

这样既能支持：
- 点击交互正向回放
- 页面值反向溯源

又不会在 MVP 阶段过重。
