# 页面初始化时接口性能问题诊断 Plan

## 1. 目标

当用户打开某个页面后，页面初始化耗时过长、首屏迟迟不出、模块长期 loading 或数据迟迟不展示时，系统需要能够定位：

1. 是否是接口性能问题导致的初始化慢
2. 是哪个接口慢
3. 是单接口慢，还是串行依赖导致整体慢
4. 是网络慢、后端慢、前端等待策略问题，还是渲染未及时收敛
5. 哪些页面症状与这些接口耗时有关

最终输出面向诊断工作台的结果，包括：

- top cause
- supporting causes
- symptoms
- 初始化阶段 domino chain
- 调试建议

---

# 2. 诊断范围

本 Plan 聚焦 **页面初始化阶段**，即用户从进入页面到页面达到“首屏可用/主要模块可见”之前的这一段时间。

## 2.1 覆盖场景
- 首次进入页面
- 路由切换进入页面
- 页面刷新后初始化
- 页面初始化自动触发的首批接口加载
- 首屏关键模块依赖的数据加载

## 2.2 暂不重点覆盖
- 用户二次操作后的查询慢
- 长列表滚动分页加载
- 页面驻留期间的后台轮询
- 用户手动点击后的局部刷新性能问题

这些可以后续作为独立 plan。

---

# 3. 核心诊断思路

页面初始化接口性能问题，不直接判断“页面慢”，而是拆成：

> **页面进入 → 初始化开始 → 首批接口发起 → 接口返回 → 页面状态变化 → 首屏完成情况**

围绕这条链路收集证据并归因。

---

# 4. 核心观测对象

需要观测 4 类对象。

## 4.1 页面初始化阶段标记
用于定义“页面初始化窗口”。

关键事件：
- route enter
- page mount
- init start
- first meaningful content / first module rendered
- init end（如能识别）

来源：
- 插件自动 route 观察
- 页面 bridge context/state 补充

---

## 4.2 初始化期接口事件
关注初始化窗口内发生的接口请求：

- 请求开始时间
- 响应结束时间
- duration
- status
- timeout / error
- 是否并行 / 串行
- 是否阻塞关键模块展示
- 是否空结果 / 结构异常

---

## 4.3 页面症状
页面初始化过程中出现的用户可见症状：

- 长时间 loading
- skeleton 持续过长
- 首屏空白
- 首模块未展示
- error toast
- empty state
- 局部模块延迟出现

---

## 4.4 业务语义上下文
用于判断哪个接口属于哪个初始化模块。

来自页面轻桥接：
- 当前 page/module
- init state: started / loading / rendered / failed
- 首屏关键模块标记
- 可选 businessAction: `pageInit`, `loadDashboard`, `loadOrderList`

---

# 5. 诊断分层目标

建议把“初始化慢”拆成 5 个可诊断层次。

## L1 页面层症状
- 页面进入后长时间不可用
- 首屏模块迟迟不出现
- loading 超时

## L2 接口层耗时
- 某一个接口超时或显著慢
- 多个初始化接口同时慢
- 接口返回失败

## L3 编排层问题
- 接口串行导致总耗时被拉长
- 非关键接口阻塞关键模块渲染
- 重复请求/重试导致初始化拉长

## L4 前端收敛层问题
- 接口返回后页面仍长时间 loading
- 数据已返回但模块未渲染
- 首屏完成信号晚于接口完成过多

## L5 后端/网络候选根因
- 慢接口集中在某域名/API 前缀
- 网关超时
- 网络连接异常
- 返回结构异常导致前端卡住

# 6. 采集方案

---

## 6.1 浏览器插件自动采集

### 页面生命周期
- route change
- document ready / SPA navigation
- 页面可见性变化
- DOM 中 loading/skeleton/empty/error 区域变化

### 网络
- 页面初始化窗口内的 fetch / XHR
- duration / status / error / timeout
- 请求开始顺序与结束顺序

### 症状
- loading 持续时间
- error toast
- 主模块空白
- 首屏关键区域是否渲染

---

## 6.2 页面轻桥接补充

建议页面在初始化阶段补这些事件：

### 上下文
```ts
setDiagnosisContext({
  appId: 'erp-console',
  module: 'dashboard',
  page: '/dashboard',
  releaseVersion: '2026.06.30',
});
```

### 初始化开始
```ts
emitDiagnosisState({
  type: 'init_started',
  module: 'dashboard',
});
```

### 关键模块开始加载
```ts
emitDiagnosisState({
  type: 'loading',
  module: 'summary-cards',
  isCritical: true,
});
```

### 关键模块渲染完成
```ts
emitDiagnosisState({
  type: 'rendered',
  module: 'summary-cards',
  isCritical: true,
  itemCount: 4,
});
```

### 页面初始化完成
```ts
emitDiagnosisState({
  type: 'init_completed',
  module: 'dashboard',
});
```

这样插件就能更清楚地圈定初始化窗口和关键模块。

# 7. 诊断窗口定义

初始化性能诊断首先要定义“窗口”。

## 7.1 开始点
满足其一即可：
- 路由进入页面
- 页面 bridge 发出 `init_started`
- 页面主容器首次挂载

## 7.2 结束点
优先级如下：
1. bridge 发出 `init_completed`
2. 首屏关键模块全部 rendered
3. 首个主要内容区渲染完成
4. 超过诊断窗口上限（例如 15s）自动截断

## 7.3 窗口内关注的请求
只分析：
- 初始化开始后发起的请求
- 且发生在窗口内
- 或虽窗口外结束但起点在窗口内的请求

避免把背景轮询和非初始化请求混进来。

# 8. 关键诊断规则

这里是 plan 的核心。  
建议 MVP 先做一组稳定规则，而不是过度智能化。

---

## R1 单关键接口超时/超慢
### 判定
- 初始化窗口内存在关键接口
- duration > slowThreshold / timeoutThreshold
- 同时关键模块长期 loading 或迟迟未 rendered

### 可能归因
- 单关键接口性能问题导致首屏慢

---

## R2 多接口并发慢
### 判定
- 初始化窗口内多个接口 duration 均较高
- 页面整体 loading 时间与这些请求结束时间重合

### 可能归因
- 初始化接口整体性能不足
- 页面首屏依赖面过大

---

## R3 串行依赖拉长初始化
### 判定
- 初始化阶段接口明显串行
- 后续请求依赖前序返回才发起
- 总耗时接近各请求耗时之和

### 可能归因
- 初始化编排串行化
- 可并行化的依赖未优化

---

## R4 非关键接口阻塞关键模块
### 判定
- 某请求不是关键模块数据
- 但页面/关键模块 rendered 明显晚于该请求完成
- 且桥接/状态表明关键模块本可更早展示

### 可能归因
- 首屏等待策略不合理
- 非关键请求阻塞首屏收敛

---

## R5 接口成功但渲染延迟过长
### 判定
- 初始化关键请求已经成功完成
- 但关键模块 rendered 晚于最后关键请求完成较多（如 > 1.5s/2s）
- 页面仍显示 loading/skeleton

### 可能归因
- 前端状态收敛慢
- 渲染阻塞
- 主线程繁忙
- 条件渲染逻辑问题

---

## R6 接口失败导致初始化不完整
### 判定
- 初始化关键请求失败/超时
- 页面出现 error / empty / blank symptom
- 关键模块未 rendered

### 可能归因
- 初始化接口失败导致首屏缺失

---

## R7 重复请求/重试导致初始化拉长
### 判定
- 同一 urlPattern 在初始化窗口内短时间多次触发
- 且后续成功或失败导致整体时长增加

### 可能归因
- 初始化逻辑重复触发
- 自动重试策略不当

---

# 9. 关键指标定义

为诊断和工作台展示，建议前端/后端统一产出以下指标。

## 9.1 页面级指标
- `initStartAt`
- `initEndAt`
- `initDuration`
- `criticalModuleRenderedCount`
- `initCompletionState`

## 9.2 接口级指标
- `requestCountInInitWindow`
- `slowRequestCount`
- `failedRequestCount`
- `timeoutRequestCount`
- `criticalRequestCount`
- `maxRequestDuration`
- `p95RequestDuration`（MVP 可不做）
- `serialChainDuration`

## 9.3 渲染收敛指标
- `lastCriticalRequestEndAt`
- `firstCriticalModuleRenderedAt`
- `finalCriticalModuleRenderedAt`
- `frontendSettleGap = finalCriticalModuleRenderedAt - lastCriticalRequestEndAt`

这个 gap 非常重要，用来判断：
- 慢在接口
- 还是慢在接口之后的前端收敛

# 10. 前端检测输出对象

建议插件前端不要直接下最终结论，而输出一组结构化 observation。

## 10.1 示例
```ts
type InitPerfObservation =
  | {
      type: 'init_window_started';
      page: string;
      timestamp: number;
    }
  | {
      type: 'critical_request_slow';
      requestId: string;
      module?: string;
      durationMs: number;
      urlPattern: string;
    }
  | {
      type: 'critical_request_failed';
      requestId: string;
      module?: string;
      status?: number;
    }
  | {
      type: 'init_serial_dependency_detected';
      chainRequestIds: string[];
      totalDurationMs: number;
    }
  | {
      type: 'frontend_settle_gap_large';
      gapMs: number;
      module?: string;
    }
  | {
      type: 'critical_module_not_rendered_after_request';
      module: string;
      requestId: string;
    };
```

这些 observation 后端很容易接。

# 11. 后端归因输出目标

后端可基于 observation + timeline + evidence 输出：

## top cause
- 页面初始化关键接口 `/api/dashboard/summary` 响应耗时 6.2s，导致首屏关键模块延迟展示

## supporting causes
- 初始化阶段 4 个接口并发耗时偏高
- 页面在最后关键请求结束后仍额外等待 1.8s 才完成渲染

## symptoms
- 首屏 summary cards 持续 loading 7.1s
- 用户在初始化阶段观察到空白卡片区域

## domino chain
- 进入页面
- 初始化开始
- 请求 summary 接口
- summary 接口耗时 6.2s
- 关键模块 loading 持续
- summary cards 延迟渲染
- 页面初始化完成时间被拉长

# 12. 前端实现计划

建议你按 3 个阶段做。

---

## Phase 1：基础初始化性能诊断
### 插件能力
- route enter 检测
- 初始化窗口粗识别
- 初始化期请求监听
- 慢请求/失败请求识别
- loading/skeleton 持续时间识别

### 输出
- 页面初始化慢的基础 observation
- 哪些请求慢
- 是否伴随 loading symptom

### 价值
- 不依赖重接入，先跑通基本诊断

---

## Phase 2：桥接增强关键模块识别
### 页面补充
- `init_started`
- `init_completed`
- `module loading/rendered`
- `isCritical`

### 提升
- 更准确圈定初始化窗口
- 更准确识别关键模块依赖的接口
- 更容易判断“接口慢”还是“前端收敛慢”

---

## Phase 3：编排与阻塞关系分析
### 增加能力
- 识别串行依赖
- 识别非关键接口阻塞关键渲染
- 识别重复请求/无效重试
- 更完整 domino chain

---

# 13. MVP 落地建议

如果你现在只想先做一版，我建议 MVP 只做以下闭环。

## 必做
- 初始化窗口识别
- 初始化期请求收集
- slow/failed/timeout 检测
- loading 长时间未结束检测
- 关键 observation 输出
- 诊断工作台展示“页面初始化慢 -> 候选慢接口”

## 推荐
- 页面桥接 `init_started/init_completed`
- 关键模块 `loading/rendered`
- 接口业务摘要（如“加载首页卡片数据”）

## 延后
- 精细关键路径算法
- 全量主线程性能分析
- 自动推断所有模块依赖
- 复杂跨页初始化建模

---

# 14. 一句话方案结论

> 页面初始化时接口性能问题诊断，应以“初始化窗口”为分析边界，围绕初始化期请求、关键模块状态和页面症状构建 observation，再由后端完成“慢接口 / 串行依赖 / 前端收敛延迟 / 初始化失败”的归因输出。
