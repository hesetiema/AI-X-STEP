# 页面值反向溯源时序图

这个图回答的问题是：

> 用户在页面上点了一个值，或者发现一个区域没展示，系统如何从 UI 反向定位到组件、状态、接口、数据转换链，并判断问题出在哪一层？

# 2. 每一步详细说明

---

## Step 1：用户选中页面上的一个值或空白区域

### 用户动作
可能是：
- 点击一个显示错误的数值
- 框选一个没显示内容的区域
- 在 inspect 模式下悬浮并选中一个元素

### 输入对象
这里的“目标”不一定有文本，也可能是：
- 空状态占位
- `--`
- `N/A`
- 隐藏的区域
- 整个组件容器

---

## Step 2：Inspect UI 发起 inspect_target 请求

前端工作台或浏览器插件把目标信息发给 Inspect Agent。

### 请求载荷示例
```json
{
  "page_url": "/order/detail?id=123",
  "dom_path": "#app .order-panel .amount",
  "selected_text": "--",
  "interaction_context_id": "ctx_001",
  "timestamp": 1718162812
}
```

---

## Step 3：Inspect Agent 获取 DOM 基础信息

这一步是“从视觉区域变成技术对象”。

### 获取内容
- CSS selector / XPath
- 文本内容
- 节点属性
- 是否可见
- 是否被 `display:none` / `visibility:hidden`
- 节点尺寸
- 是否在视口内
- 是否命中 fallback class

### 目的
先确定：
- 是真的没渲染
- 还是渲染了但隐藏了
- 还是显示了 fallback 值

---

## Step 4：DOM/Component Mapper 定位所属组件

这是很关键的一步。

### 目标
把 DOM 节点映射到：
- React Fiber 节点 / Vue Component 实例
- 组件名
- 文件位置
- props snapshot
- state snapshot
- 父组件路径

### 输出示例
```json
{
  "component": "OrderAmount",
  "file": "src/pages/order/detail/OrderAmount.tsx",
  "props": {
    "amount": null,
    "currency": "CNY"
  },
  "render_expression": "formatCurrency(amount)"
}
```

---

## Step 5：Inspect Agent 发起反向溯源请求

一旦确定了页面元素属于哪个组件，系统就有能力继续反查：

- 这个展示值来源于哪个 props / state
- props 又来自哪里
- 是否来自 store / query / parent props
- 是否经过 transform

---

## Step 6-8：Lineage Resolver 查询运行时 Trace

这一步主要找“真实发生过什么”。

### 查询内容
- 最近一次 render 事件
- render 前后的 props
- 相关 selector 输出
- 最近 store 更新
- 最近请求响应
- DOM patch 历史
- 当前区域对应的 interaction trace

### 价值
运行时证据能回答：
- 最近一次组件拿到的值是什么
- 是什么时候变成空的
- 变空前后是否发生请求
- 是否有重渲染

---

## Step 9-10：查询静态代码图谱

静态图谱负责补足“理论依赖关系”。

### 静态图谱可提供
- JSX/模板中这个值绑定了哪个表达式
- `formatCurrency(amount)` 里的 `amount` 来自 props 还是 selector
- selector 又读了哪个 store path
- store path 对应哪个 action / API adapter
- API adapter 对应哪个 response field

### 举例
静态分析得到候选链路：

```text
OrderAmount.tsx
  -> props.amount
  -> from useOrderDetail()
  -> from orderStore.currentOrder.amount_cent
  -> from normalizeOrder(response.data)
  -> from GET /api/order/detail response.data.amount_cent
```

---

## Step 11：合并静态 + 运行时，形成反向血缘路径

这是平台最核心的一步。

### 反向血缘路径典型长相
```text
DOM Text "--"
← render expression: formatCurrency(amount)
← component prop: amount = null
← selector/store: orderStore.currentOrder.amount_cent = null
← API adapter: normalizeOrder(response.data).amount_cent = null
← API response: GET /api/order/detail => response.data.amount_cent missing
```

或者：

```text
DOM Text "--"
← render expression: formatCurrency(amount)
← component prop: amount = 0
← formatter: if (!amount) return "--"
← API response: amount_cent = 0
```

这个例子就很典型地说明：
- 接口没问题
- 问题出在前端 falsy 判断

---

## Step 12-13：规则引擎做确定性检查

这一层用于先给出高确定性的判断。

### 常见规则
#### 规则 A：字段缺失
- 接口响应里没有该字段
- 适配层输出 undefined

#### 规则 B：falsy 误判
- `if (!value)` 误伤 `0`
- `""` / `false` / `[]` 被误判为空

#### 规则 C：fallback 吞值
- formatter 内部兜底过强
- `return value || "--"`

#### 规则 D：状态更新未触发展示更新
- store 变了
- 组件没刷新
- memo/依赖数组错误

#### 规则 E：条件渲染阻断
- `value && <Text/>`
- 0 时直接不渲染

#### 规则 F：旧缓存
- query cache 未失效
- 组件读了旧值

---

## Step 14-15：AI 生成解释和建议

AI 的输入不是原始日志，而是结构化诊断上下文。

### 输入示意
```json
{
  "displayed_value": "--",
  "component": "OrderAmount",
  "render_expr": "formatCurrency(amount)",
  "api_field_value": 0,
  "store_value": 0,
  "formatter_output": "--",
  "rule_hits": ["falsy_misjudge", "fallback_overwrite"],
  "confidence": 0.92
}
```

### AI 输出示意
- 该页面值来自订单详情接口的 `amount_cent` 字段
- 接口和 store 中值均为 `0`
- 问题出在前端 `formatCurrency` 中使用 `if (!amount)` 判断空值，导致 `0` 被误判为空并展示为 `--`
- 建议改为 `amount == null` 判空，并补充 `0` 场景单测

---

## Step 16：诊断报告 UI 展示

最终给用户看到的，不应该是一堆日志，而应该是：

### 诊断结果结构
1. **当前展示状态**
   - 页面显示：`--`
   - 预期值：`¥0.00`

2. **反向血缘路径**
   - DOM
   - 组件
   - props
   - selector/store
   - API 字段

3. **根因排序**
   - falsy 误判：92%
   - 接口字段缺失：3%
   - 渲染未刷新：5%

4. **证据链**
   - 接口返回值：0
   - store 值：0
   - formatter 输出：`--`

5. **修复建议**
   - 修改 formatter 判空逻辑
   - 增加 0 值测试
   - 增加字段显示监控

# 3. 异常分支时序图

下面我再补一版更贴近异常判断的分支图。

---

## 3.1 分支图：值没显示，到底断在哪一层

```text
用户选择页面异常区域
        │
        ▼
Inspect Agent 定位 DOM / 组件
        │
        ▼
是否存在对应 DOM 节点？
   ├─ 否 → 可能是条件渲染未进入 / 组件未渲染
   └─ 是
        │
        ▼
DOM 是否可见？
   ├─ 否 → CSS隐藏 / 折叠 / 遮挡
   └─ 是
        │
        ▼
组件最近是否渲染？
   ├─ 否 → 上游状态变化未触发刷新
   └─ 是
        │
        ▼
组件 props / state 中是否有目标值？
   ├─ 否 → 上游状态层/父组件传值异常
   └─ 是
        │
        ▼
render expression 输出是否为空/fallback？
   ├─ 是 → formatter / 条件表达式 / fallback 逻辑异常
   └─ 否
        │
        ▼
store/query 中是否有值？
   ├─ 否 → store层未写入/缓存陈旧
   └─ 是
        │
        ▼
API response 中是否有字段？
   ├─ 否 → 接口字段缺失/契约漂移
   └─ 是
        │
        ▼
最终归因:
- 接口问题
- 状态同步问题
- transform问题
- 渲染问题
- DOM展示问题
```

# 4. 更适合开发实现的“服务时序版”

如果你是给工程团队讨论，这版会更实用。

```text
User
  │
  │ select target
  ▼
Browser Extension / Inspect Panel
  │
  │ inspect(domNodeId, pageContext)
  ▼
Inspect Service
  │
  ├── query DOM Mapper
  │      └── return component instance + render binding
  │
  ├── query Trace Service
  │      └── return recent render/state/request timeline
  │
  ├── query Static Graph Service
  │      └── return code dependency candidates
  │
  ├── call Lineage Resolver
  │      └── merge runtime + static path
  │
  ├── call Rule Engine
  │      └── return deterministic findings
  │
  ├── call AI Diagnosis Service
  │      └── return explanation + fix suggestions
  │
  ▼
Diagnosis API Response
  │
  ▼
Inspect Panel / Report UI
```

# 5. 这个时序图对应的核心接口建议

如果后面要做工程设计，这几个接口大概率会有。

---

## 5.1 inspect target
```http
POST /api/inspect/target
```

### 入参
- page_url
- dom_selector / dom_node_id
- selected_text
- session_id
- interaction_context_id

### 出参
- component info
- current dom snapshot
- render binding summary

---

## 5.2 reverse lineage
```http
POST /api/lineage/reverse-trace
```

### 入参
- target component
- target field / dom node
- timestamp window
- page session id

### 出参
- lineage path
- runtime evidence
- static candidates
- suspected breaks

---

## 5.3 diagnose anomaly
```http
POST /api/diagnosis/anomaly
```

### 入参
- lineage path
- runtime snapshots
- rule hits

### 出参
- root cause ranking
- evidence list
- suggestions

# 6. MVP 里最小可实现版本

如果你只做 MVP，这个时序图其实可以收缩成下面这样：

```text
用户点页面值
   ↓
插件拿到 DOM + 组件名
   ↓
查最近一次 render props
   ↓
查最近一次 store 值
   ↓
查最近一次 API response 字段
   ↓
规则判断是哪层出问题
   ↓
AI 用自然语言输出诊断结论
```

### MVP 阶段先不必做到
- 完整静态 AST 血缘
- 复杂跨层多候选路径评分
- 全量时序回放

先把下面三件事做稳就已经很有价值：
1. 能定位组件
2. 能定位 store/API/formatter
3. 能输出“问题在接口/转换/渲染哪层”
